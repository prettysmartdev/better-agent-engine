# Work Item: Feature

Title: harness launchers
Issue: issuelink

## Summary:
- This work item introduces a new concept to bae: launchers

Launchers are Dockerfile templates embedded with small pieces of code (either TypeScript, Rust, or both depending on the type) which users can optionally load their custom agent harnesses into in order to be triggered or launched in specific ways.

Three launchers will be available to start:

1) Schedule launcher: a small Rust binary contained within a Docker image which is given 1 or more custom agent harnesses that the user builds (and includes in the Docker build by extending the base image provided by bae). A config file (bae-schedules.toml) then allows the user to define which agents are included in the image, how each one should be launched (env vars, command line arguments, etc), and a crontab for each to determine when each one should be triggered. The schedule launcher application (i.e. the Rust binary) reads the config file, sets up cron-like timers/schedules to trigger the configured agents on the provided schedule.

2) API launcher: a different small Rust binary which is also built into a base Docker image which allows the user to package 1 or more custom agent harnesses via a Dockerfile that extends the base image. A config file (bae-api.toml) defines which agents are present, the format of the API POST request that should be enabled to trigger each one, and instructions on how to launch each agent given the POST payload (use JSONSchema or similar to define POST request bodies and shell templates to define how fields from the POST payload get applied to command line arguments/env vars/etc to launch each agent. The API server should start up, expose a configurable port, and serve a JSON HTTP API per the config file to allow triggering an agent when the appropriate endpoint is hit.

3) Webapp launcher: a TypeScript webapp (use same frameworks/conventions as MAX) plus the SAME Rust binary from launcher #2 are packaged into a Docker base image. When the user provides a `bae-app.toml` (similar in format to `bae-api.toml`), the API server and webapp work together to serve a web interface that allows the user to see which agents are configured, open detail pages for each one, and enter into a chat interface where either freeform prompts OR pre-defined prompts from the config file can be sent to one of the configured agents via the Rust API server, triggering in a similar fashion to the API launcher in #2 but with a nice user-friendly webapp. The interface should have a grid/card-like interface "home screen" to show all available agents, and a nice familiar chatbot-like view to interact with a specific agent.

For all 3 options, the user should be able to create a Dockerfile that uses FROM: {one of the 3 base images} and include their custom harness code or built binaries, the appropriate config file, and build that Dockerfile into a simple image that can be deployed and controlled with env vars. There should be no local persistence for V1, simplest possible config, and user-friendly examples of Dockerfiles that can be used to build their own version of the launcher images. The base images should be built by GitHub actions and included whenever a vx.y.z release is triggered and published to GHCR alongside the standard and MAX variants of the baesrv images.

## User Stories

### User Story 1:
As an: Agent Developer

I want to:
extend the `bae-launcher-schedule` base image with my own custom agent harness (a binary or script I build, using any language or none of bae's SDKs at all) and a `bae-schedules.toml` naming when it should run, with what arguments and environment variables

So I can:
get my harness running on a recurring cron schedule inside a single, deployable container, without writing any scheduling, timer, or process-supervision code myself

### User Story 2:
As an: Agent Developer / Integrator

I want to:
extend the `bae-launcher-api` base image, describe one or more agents in `bae-api.toml` (each with a JSON Schema for its trigger payload and a template mapping payload fields to command-line arguments/env vars), and have a JSON HTTP endpoint appear automatically for each one

So I can:
let an external system (a webhook, a CI job, another service) trigger my harness over HTTP with a validated request body, without writing an HTTP server, request validation, or argument-templating code myself

### User Story 3:
As a: Non-technical stakeholder / on-call operator

I want to:
open the `bae-launcher-webapp` image's web dashboard, see a card for every configured agent, open one, and send it either a free-form prompt or a pre-defined prompt from a chat-style view

So I can:
trigger and interact with a packaged agent harness from a browser, without knowing curl, the agent's JSON Schema, or the container's command line at all

## Implementation Details:

This work item adds three new, independently-buildable **base images** — meant to be `FROM`-extended by an end user's own Dockerfile, not run standalone in production — plus the GitHub Actions/Makefile plumbing to publish them to GHCR on every release alongside the existing standard and `bae-max` images. It splits into six pieces: (A) shared conventions all three launchers follow; (B) the schedule launcher; (C) the API launcher; (D) the webapp launcher, which reuses the API launcher's binary unmodified; (E) CI/release integration; (F) examples and docs.

**Deliberately decoupled from `server/`, `client-rust/`, `client-typescript/`, `client-python/`, and `baectl/` — but free to depend on each other.** A launcher's job is to spawn a configured child process with templated env vars/args on a trigger (cron fire or HTTP POST) — it has no bae-specific awareness and never imports any client SDK or talks to a `baesrv` instance. The harness the user packages *may* itself be built with a bae client SDK and may read ordinary bae env vars like `BAE_CLIENT_KEY`/`server_url` (`docs/guides/building-a-client.md`), but that is entirely the harness's business, opaque to the launcher. This mirrors the reasoning `aspec/work-items/0004-baectl-cli.md` gives for why `baectl` doesn't depend on `client-rust`, and `aspec/work-items/0007-max-webapp.md` gives for why `max/server` doesn't depend on `client-typescript` — each new component only takes on the dependencies its own scope actually needs. That boundary is drawn around `launchers/` as a whole, not around each launcher individually: `launchers/schedule/` and `launchers/api/` **do** share code with each other (via `launchers/core/`, section A) — sharing within the launchers family is encouraged; reaching outside it is not.

### A. Shared conventions

- **One or more agents per image, in all three launchers — a first-class requirement, not an incidental config detail.** All three config files (`bae-schedules.toml`, `bae-api.toml`, `bae-app.toml`) use a `[[agents]]` **array of tables** from V1 day one, never a single top-level `[agent]` block — a config with exactly one entry and a config with twenty are the same shape, so there is no "upgrade path" a user ever has to hit. A single Dockerfile extending a single base image is expected to bundle a whole related family of harnesses (e.g. `nightly-report` + `weekly-digest` + `on-demand-cleanup` all in one `bae-schedules.toml`; `summarize` + `translate` + `triage` all in one `bae-api.toml`/`bae-app.toml`). Concretely, this means: each launcher process runs **N independent agents concurrently in one container** — N cron jobs (schedule launcher) or N routes on one shared router/port (API/webapp launcher) — with no built-in cap on N in V1. `name` uniqueness (already a fatal startup error, see below) is what makes this safe: it is simultaneously the cron-log identifier, the URL path segment, and the webapp's card key, so one namespace covers all three launchers' multi-agent addressing.
- **Per-agent log-line attribution.** Because a single container commonly runs several agents' output concurrently (independent cron fires interleaving, or concurrent HTTP triggers hitting different agents at once), every line of a child process's captured stdout/stderr is prefixed with its agent's `name` before being forwarded to the launcher's own stdout/stderr (e.g. `[nightly-report] backing up database...`) — without this, `docker logs` on a multi-agent image would be an unattributable interleaved mess the moment two agents' invocations overlap in time. This is new behavior with no direct precedent elsewhere in the codebase (`server/src/engine/{mcp,sandbox}.rs` each supervise a single logical child at a time), added specifically because this work item's images are multi-agent by design.
- **One top-level parent folder, `launchers/`, holding all three components as subfolders** — `launchers/schedule/` (section B), `launchers/api/` (section C), and `launchers/webapp/` (section D) — instead of three separate top-level directories, so the repo root gains one new entry for this whole feature rather than three. Cargo's/npm's package `name` field is independent of its directory location (`client-rust/` already produces a differently-named crate, `bae-rs`, per `aspec/architecture/design.md` Component 2), so nesting under `launchers/` requires no renaming anywhere else: the crates, binaries, Dockerfiles, Makefile image targets, GHCR tags, and env vars all keep their existing flat `launcher-schedule`/`launcher-api`/`launcher-webapp` identifiers — only the three components' own source paths gain the `launchers/` prefix.
- **New Rust crates `launchers/schedule/` (package `launcher-schedule`, binary `baesched`) and `launchers/api/` (package `launcher-api`, binary `baeapi`)**, each standing alone exactly like every existing Rust component — own `Cargo.toml`, own `Makefile` with the standard `build`/`test`/`lint`/`fmt`/`clean` verbs, `publish = false` (never released to crates.io — ships only inside the launcher images, version tracks the image tag, same posture `aspec/devops/cicd.md` already documents for `baectl` and `max`). No root Cargo workspace exists today (confirmed: no `/Cargo.toml`) and this work item does not add one — each crate keeps its own `target/`, matching `baectl/Cargo.toml`'s and `server/Cargo.toml`'s independent-crate pattern; `launchers/` itself is a plain filesystem grouping, not a Cargo workspace root.
- **Shared library crate `launchers/core/` (package `launcher-core`), a workspace-free path dependency of both `launchers/schedule/` and `launchers/api/`.** `launcher-schedule` and `launcher-api` need the same underlying primitive — "resolve `${ENV_VAR}` in config, spawn a `tokio::process::Command` with templated args/env, stream/capture and log-prefix its output" — closely enough that duplicating it would mean fixing the same bug twice. Unlike the earlier `Dockerfile.max`-style "small duplication over coupling" trade-off (which is about *not* coupling two independently-shipped *images'* build stages together), this is two crates that are *already* siblings under one `launchers/` parent, shipped as part of one feature, maintained together — coupling them to each other costs nothing extra and removing the duplication is a straightforward win. **Launchers may depend on each other; they still depend on nothing outside `launchers/`** — `launcher-core` itself never imports `server/`, `client-rust/`, `client-typescript/`, `client-python/`, or `baectl/`, preserving the decoupling this section opens with. A plain Cargo path dependency (`launcher-core = { path = "../core" }` in both `launchers/schedule/Cargo.toml` and `launchers/api/Cargo.toml`) needs no root workspace to work — `launchers/` stays "a plain filesystem grouping, not a Cargo workspace root" exactly as the bullet above states; each launcher Dockerfile's build stage `COPY`s `launchers/core/` alongside its own crate (`launchers/schedule/` or `launchers/api/`), mirroring how the root `Dockerfile` already `COPY`s `server/` and `baectl/` into separate subdirectories of one build stage for the same reason (so a relative path dependency resolves inside the build context).
  - **`launcher-core`'s scope, precisely**: (1) `SpawnSpec { name, command, args, env, working_dir }` — the fully-resolved, post-templating shape both crates convert their own richer per-agent config into before handing off to shared code (each crate keeps its own config struct — `ScheduleAgentConfig` with `schedule: String`, `ApiAgentConfig` with `request_schema`/`env_template`/`arg_template` — since those genuinely differ; only the *resolved, ready-to-spawn* shape is shared). (2) `resolve_env_refs(&HashMap<String, String>, environ: &dyn Fn(&str) -> Option<String>) -> Result<HashMap<String, String>, LauncherError>` — the `${VAR}` resolution helper (see below), injectable-environment for testability, mirroring `max/server/src/config.ts`'s `loadConfig(env, ...)` pattern of taking the environment lookup as a parameter rather than reading `std::env` directly. (3) `spawn_and_stream(&SpawnSpec) -> impl Stream<Item = LogLine>` — `tokio::process::Command` + `kill_on_drop(true)`, concurrently draining stdout/stderr, yielding length-capped, agent-name-prefixed lines (the log-line-attribution bullet above) that both the scheduler (forwards to its own stdout) and the API server (forwards to its own stdout **and** streams to the HTTP response body, section C) consume identically. (4) `validate_unique_names<'a>(names: impl Iterator<Item = &'a str>) -> Result<(), LauncherError>` — the duplicate-`name` startup check both configs need. (5) `init_logging(default_filter: &str)` — one `tracing`/`tracing-subscriber` setup function reading `BAE_LOG`, so both binaries initialize logging identically. (6) `LauncherError` with the shared `exit_code()` mapping (usage error → 2, runtime error → 1), matching `aspec/uxui/cli.md`'s convention and `server/src/config.rs`'s `ConfigError::exit_code()` precedent. Everything HTTP-specific (axum routing, JSON Schema validation, the streamed-response body) and everything cron-specific (`tokio-cron-scheduler` job registration, the overlap-skip policy) stays in `launchers/api/` and `launchers/schedule/` respectively — `launcher-core` has no knowledge of HTTP or cron at all.
- **`${ENV_VAR}` secrets convention** (implemented once, in `launcher-core`'s `resolve_env_refs` — point (2) above): both `bae-schedules.toml`'s `env` map and `bae-api.toml`'s `env_template` values may contain `${VAR}` tokens, resolved from the launcher process's own environment immediately before spawning the child — never logged, never written anywhere — the exact pattern `server/src/config_file.rs` already documents for `bae-config.toml`'s `headers`/`auth_token` fields ("the raw tokens are preserved... and resolved immediately before connecting/calling, via `resolve_tokens`, so the resolved secret is never persisted"). A referenced env var that is unset at spawn time is a **hard failure for that invocation** (logged, not silently substituted as an empty string) — silently degrading a missing secret to `""` is exactly the failure mode the existing convention's secret-handling discipline is written to avoid.
- **Config loading semantics mirror `bae-config.toml`'s, with one deliberate difference.** Like `server/src/config_file.rs`, a *missing* config file is not fatal — the launcher starts with zero configured agents and logs a warning (an image built from a launcher base with no config dropped in yet should still start cleanly, e.g. during early `docker build` iteration). Unlike `bae-config.toml`, a config file that **exists but is malformed** (bad TOML, an invalid cron expression, an invalid JSON Schema, or a duplicate agent `name`, caught by `launcher-core`'s `validate_unique_names` — point (4) above) is a **fatal startup error**, exit code 2 (usage error, matching `aspec/uxui/cli.md`'s existing `baesrv`/`baectl` exit-code convention: 0 success, 1 runtime error, 2 usage error, also shared via `launcher-core`'s `LauncherError` — point (6) above) — a launcher with a broken config has no useful degraded mode to fall back to (unlike the server, which can usefully run with an empty MCP registry).
- **Every launcher takes its config path via one env var with a per-image default**, following `BAE_CONFIG`'s exact flag-then-env precedence shape: `baesched` reads `BAE_SCHEDULES_CONFIG` (default `/etc/bae/bae-schedules.toml`); `baeapi` reads `BAE_LAUNCHER_API_CONFIG` (default differs per image — `/etc/bae/bae-api.toml` baked into `Dockerfile.launcher-api`'s `ENV`, `/etc/bae/bae-app.toml` baked into `Dockerfile.launcher-webapp`'s `ENV` — same binary, same env var name, different default value per base image, so a user's own Dockerfile only has to `COPY` the right filename to the default path and never has to touch the env var at all).
- **Logging**: `BAE_LOG` env var (default `info`), same env var name as `server/src/config.rs`'s `DEFAULT_LOG`, initialized identically in both binaries via `launcher-core`'s `init_logging` (point (5) above) — every child process's captured stdout/stderr is forwarded to the launcher's own stdout/stderr, agent-name-prefixed (never written to a file — "no local persistence for V1" applies to run history/output, not just config), so `docker logs`/`container logs` is the only place to look, exactly like every other component in this project.
- **A hung or crashed child agent invocation never takes down the launcher process itself** — deliberately the opposite of `bae-max`'s documented "either process dying takes the whole container down" exception (`aspec/devops/infrastructure.md`). A launcher may be running many independent agents/schedules; one misbehaving agent must not stop the scheduler's other timers or the API server's other routes from working. Call this out explicitly wherever this work item's infra doc update lands, since it is a deliberate divergence from the codebase's other documented multi-process container, not an inconsistency.

### B. Schedule launcher (`launchers/schedule/`, binary `baesched`, image `bae-launcher-schedule`) — depends on `launchers/core/`

- `bae-schedules.toml` shape — **one image, multiple independently-scheduled agents**:
  ```toml
  [[agents]]
  name     = "nightly-report"
  command  = "/usr/local/bin/nightly-report-harness"
  args     = ["--mode", "report"]
  env      = { AGENT_MODE = "scheduled", API_TOKEN = "${MY_HARNESS_TOKEN}" }
  working_dir = "/app"
  schedule = "0 0 3 * * *"   # 6-field cron: sec min hour day month day-of-week

  [[agents]]
  name     = "weekly-digest"
  command  = "/usr/local/bin/weekly-digest-harness"
  args     = ["--mode", "digest"]
  schedule = "0 0 8 * * MON"
  ```
  `name` must be unique within the file (fatal startup error otherwise — it is the only identifier in logs and metrics, and the log-prefix key described in section A). `working_dir`/`env` are optional; `command` is resolved against `PATH` if not absolute, same resolution `tokio::process::Command::new` already gives for free. Each agent's `command` is typically a distinct binary/script the user's own Dockerfile `COPY`s to its own path — the base image places no constraint on how many harnesses are bundled or where they live, only that each configured `command` path exists in the final image.
- **Cron engine: `tokio-cron-scheduler`** (tokio-native, actively maintained, standard 6-field cron syntax including seconds) — registers one job **per `[[agents]]` entry** at startup, so an N-agent config yields N independently-scheduled jobs in the same process; each firing spawns the templated `tokio::process::Command` and awaits it in its own task.
- **Overlap policy (V1): skip, don't queue — scoped to the *same* agent only.** If a schedule fires while the previous invocation of that *same* agent is still running, the new fire is skipped and logged as a warning (`agent "<name>" skipped: previous invocation still running`) rather than piling up concurrent/queued runs — the simplest correct behavior for V1, and consistent with "simplest possible config." **Different agents in the same config always run concurrently with each other with no coordination or serialization** — `nightly-report` and `weekly-digest` firing at the same instant is normal, expected, and unthrottled in V1 (no shared resource limit across agents). A future work item may add a configurable `overlap_policy` (`skip` | `concurrent` | `queue`) and/or cross-agent concurrency limits; not built here.
- **No HTTP surface at all**, by design (the summary describes only "cron-like timers/schedules," no port). Liveness is purely process-level (`docker ps` / the container's own exit status) — no `/healthz` route to add, since there is no HTTP listener to serve it from. State this explicitly in docs so it isn't mistaken for an oversight relative to the other two launchers.
- Signal handling: `baesched` traps `SIGTERM`/`SIGINT`, stops accepting new fires, and gives any in-flight child invocation a bounded grace period (reuse `server/src/config.rs`'s `DEFAULT_SHUTDOWN_SECS`-style pattern — a new `BAE_SCHEDULES_SHUTDOWN_TIMEOUT`, default 30s) before force-killing it, then exits 0.

### C. API launcher (`launchers/api/`, binary `baeapi`, image `bae-launcher-api`) — depends on `launchers/core/`

- `bae-api.toml` shape — **one image, multiple agents, one shared port**:
  ```toml
  [server]
  addr = "0.0.0.0:9090"        # optional; overridden by BAE_LAUNCHER_API_ADDR if set

  [[agents]]
  name    = "daily-digest"     # becomes POST /agents/daily-digest/trigger
  command = "/usr/local/bin/daily-digest-harness"
  args    = ["--mode", "digest"]
  working_dir = "/app"

  [agents.request_schema]      # JSON Schema the POST body must satisfy
  type = "object"
  required = ["prompt"]
  [agents.request_schema.properties.prompt]
  type = "string"

  [[agents.env_template]]      # schema field -> child-process env var
  field = "prompt"
  env   = "AGENT_PROMPT"

  [[agents.arg_template]]      # schema field -> extra CLI arg (appended after `args`)
  field = "priority"
  flag  = "--priority"

  [[agents]]
  name    = "translate"        # becomes POST /agents/translate/trigger
  command = "/usr/local/bin/translate-harness"

  [agents.request_schema]
  type = "object"
  required = ["text"]
  [agents.request_schema.properties.text]
  type = "string"
  ```
- **HTTP framework: `axum`**, the same framework `baesrv` already uses for both its client and admin ports (`aspec/architecture/design.md`'s architecture diagram: `"REST + JSON-RPC /api/v1 (axum)"`) — no new HTTP-stack dependency introduced into the project.
- **Routing**: one `POST /agents/{name}/trigger` route **per configured agent, all registered on one shared `axum::Router` bound to one port** — an N-agent config exposes N independent endpoints from a single `baeapi` process with no per-agent process isolation (each trigger just spawns its own `tokio::process::Command`, so two agents' invocations never contend for a shared listener or interfere with each other's request handling), plus two fixed routes present on every `baeapi` instance regardless of how many agents are configured: `GET /healthz` (unauthenticated liveness, matching the existing `baesrv` convention referenced in `aspec/work-items/0007-max-webapp.md`'s test plan) and `GET /_launcher/agents` (read-only introspection listing every configured agent — see section D, shared with the webapp launcher).
- **Request validation: the `jsonschema` crate** (pure-Rust JSON Schema validator, no TLS/native dependency, so it carries no static-linking risk analogous to `reqwest`'s rustls-vs-OpenSSL concern in `baectl/Cargo.toml`) validates the POST body against `agents.request_schema` before templating. A schema-invalid body returns `400` with an RFC 7807 problem-details body (the same error-body shape `baectl/src/error.rs` already maps admin-API errors into — reused here for a consistent error-body convention across every bae-published binary, not invented fresh) listing the validator's failure path(s); it never reaches the child process.
- **Templating**: each `env_template`/`arg_template` entry copies one validated field's JSON value into an env var or an appended CLI flag+value on the spawned command, using the same `${VAR}` env-passthrough resolution described in section A for `env`. Fields not referenced by any template entry are accepted (schema-valid) but simply unused — a deliberate simplification over building a general templating language.
- **Streaming response, no persistence.** `POST /agents/{name}/trigger` spawns the child and streams its interleaved stdout/stderr back to the caller as the response body arrives (`axum::body::Body::from_stream` over the child's piped output, chunked transfer), finishing with a trailing NDJSON line reporting the exit code — the simplest way to give a caller (`curl --no-buffer`, or the webapp's chat view in section D) live output with **zero** server-side run history, matching "no local persistence for V1." A caller that doesn't want streaming just buffers the full response body, which still works with `curl`'s default behavior.
- **Auth: optional, off by default in V1**, exactly the posture the admin port itself shipped with before `aspec/work-items/0004-baectl-cli.md` added auth (`docs/reference/admin-api.md` "there is no auth on this port initially because it is localhost-only" — this launcher's port is *not* loopback-only, so the parallel is imperfect, but the "ship the simplest thing, harden in a follow-up" precedent is directly on point). New env var `BAE_LAUNCHER_API_TOKEN`: unset by default (open port, loud startup warning logged); when set, every `/agents/*` route (not `/healthz`, not `/_launcher/agents`) requires `Authorization: Bearer <token>`. Document prominently, per `aspec/architecture/security.md`'s existing guidance, that this port belongs behind the same TLS-terminating reverse proxy every other bae port already requires before touching a real network.
- **Port**: `BAE_LAUNCHER_API_ADDR`, default `0.0.0.0:9090` — distinct from `baesrv`'s `8080`/`8081` so a launcher and a `baesrv`/`bae-max` container can coexist on the same host/network without a collision, should a harness talk to both.

### D. Webapp launcher (`launchers/webapp/`, image `bae-launcher-webapp`) — reuses `baeapi` unmodified

- **The exact same `baeapi` binary from section C, byte-for-byte** — the summary's "the SAME Rust binary from launcher #2" — copied into this image from the identical `launchers/api/` build output. No new Rust code. This makes `bae-launcher-webapp` structurally *simpler* than `bae-max`: one process, not two — no dual-process entrypoint script, no signal-forwarding, no "either process dying kills the container" logic needed (contrast `docker/bae-max-entrypoint.sh`), because there is no second backend process here at all.
- `bae-app.toml` is `bae-api.toml`'s schema **plus** optional presentation fields `baeapi` already parses (and simply ignores when running as the plain API launcher): `display_name`, `description`, `icon` (an emoji or image URL) per agent, `chat_input_field` (which `request_schema` field the chat box's free-text input fills — default `"prompt"`), and a `[[agents.prompts]]` array of `{label, prompt}` pairs for the pre-defined-prompt buttons the summary asks for.
- **New TypeScript package `launchers/webapp/web/`** — Vite + React, scaffolded and structured identically to `max/web/` (`aspec/work-items/0007-max-webapp.md` section A: same `npm create vite@latest -- --template react-ts` starting point, same build-to-static-`dist/` output, `"private": true`, package name `@prettysmartdev/bae-launcher-webapp-web`). **No `launchers/webapp/server/` Node backend** — unlike MAX, the frontend talks directly to `baeapi`'s own JSON routes (`GET /_launcher/agents`, `GET /_launcher/agents/{name}`, `POST /agents/{name}/trigger`) via `fetch`, since `baeapi` already serves plain, unauthenticated-by-default JSON with no admin-key/session-key indirection to hide from the browser the way MAX's admin/client key material had to be — there is nothing here that needs a trusted backend to broker.
- `baeapi` gains one more optional behavior, gated by a new env var `BAE_LAUNCHER_WEBAPP_STATIC_DIR` (set by `Dockerfile.launcher-webapp`'s `ENV`, default `/usr/local/lib/launchers/webapp/web/dist` — mirroring the source path the same way `Dockerfile.max`'s `/usr/local/lib/max/web/dist` mirrors `max/web/dist`; left **unset** in `Dockerfile.launcher-api`, so the plain API launcher image never serves any static assets or UI): when set, `baeapi` additionally serves that directory as a static SPA (via `tower-http`'s `ServeDir`, a crate already precedented in this ecosystem's typical axum stack) at `/`, falling back to `index.html` for client-side routing.
- **`GET /_launcher/agents`** (introspection, present on every `baeapi`, not just the webapp image) returns each configured agent's `name`, `display_name`, `description`, `icon`, `request_schema`, `chat_input_field`, and `prompts` — **never** `env`, `env_template`'s env-var *names* are fine to expose (they're not secrets, just wiring) but resolved `${VAR}` *values* are never included, never computed for this route at all. `GET /_launcher/agents/{name}` returns the single-agent detail shape the webapp's detail page renders.
- **UI**: a card-grid home screen — **one card per configured agent**, scaling from one card up to however many `[[agents]]` entries `bae-app.toml` defines (icon, display name, description per `GET /_launcher/agents`) — linking to a per-agent detail page with a chat-style view: a message list, a free-form text input, and a row of buttons for each configured `prompts` entry. Sending either kind of message `POST`s to `/agents/{name}/trigger` with `{[chat_input_field]: <text>}` and renders the streamed response (section C) as it arrives, token-chunk by token-chunk, in a chat bubble — **no client- or server-side chat history persistence**, a page refresh clears the transcript, matching "no local persistence for V1" without qualification.
- **Empty state**: an image built from `bae-launcher-webapp` with no `bae-app.toml` (or an empty `[[agents]]` list) shows a home screen explaining what an agent is and how to configure one, rather than a bare blank grid — the same "every empty list explains what the resource is" convention `aspec/work-items/0007-max-webapp.md`'s Profiles tab already establishes for this project's UI.

## Edge Case Considerations:

- **Missing config file at startup**: both `baesched` and `baeapi` start with zero configured agents and log a warning — not fatal, so an image built mid-iteration (harness copied in, config not yet added) still starts cleanly. See section A.
- **Malformed config that exists** (bad TOML, invalid cron expression, invalid JSON Schema, duplicate `name` across `[[agents]]` entries): fatal at startup, exit code 2 — a broken config has no useful degraded mode, unlike a missing one. Duplicate-`name` detection matters specifically *because* multi-agent configs are the norm, not the exception — a copy-pasted `[[agents]]` block with an unchanged `name` is the most likely way this error actually gets hit.
- **Overlapping cron fires** for the same agent: the later fire is skipped and logged, never queued — see section B. A very short interval combined with a slow harness can mean an agent effectively "misses" most of its scheduled fires; document this plainly rather than silently masking it.
- **Two or more different agents' schedules/triggers firing at the same instant** (schedule launcher: independent cron expressions that happen to coincide; API/webapp launcher: concurrent HTTP requests to two different `/agents/{name}/trigger` routes): both run fully concurrently with no throttling, coordination, or shared resource limit across agents in V1 — this is expected, not a race condition to guard against, and must not be conflated with the same-agent overlap-skip policy in section B (which applies only within one agent's own schedule).
- **Log output from several agents interleaving in `docker logs`**: every forwarded stdout/stderr line is prefixed with its agent's `name` (section A) specifically so a multi-agent image's combined log stream stays attributable; a launcher that only ever ran one agent would not need this, but since 1-agent and N-agent configs share the exact same code path, the prefix is always applied, even for a single-agent image.
- **A user's own Dockerfile `COPY`s two harnesses to the same path**, or a `command` in one `[[agents]]` entry accidentally points at another agent's binary: this is a user configuration error the launcher cannot detect (it has no notion of which binaries "belong" to which agent) — document that each agent's `command` should resolve to a distinct, correct path, the same way the base image's own comments already warn against overriding `ENTRYPOINT`.
- **Many agents configured in one image (tens, not two or three)**: no hard cap in V1 for any of the three launchers, though the webapp launcher's card grid should remain usable at that scale (basic wrapping/scrolling, not a fixed 2x2 layout) and the schedule launcher's `tokio-cron-scheduler` registration/signal-handling loop must remain O(N) startup/shutdown, not something that degrades badly as N grows — not a documented hard limit, just a sanity bound worth a comment where the job-registration loop lives.
- **A hung or crashed child agent process** never crashes or blocks the launcher itself — a deliberate divergence from `bae-max`'s "either process dying takes the container down" model, since a launcher typically hosts multiple independent agents/schedules where one bad actor should not take the rest down. See section A.
- **`${ENV_VAR}` referenced in `env`/`env_template` but unset** at spawn time: that specific invocation fails loudly (logged; for the API/webapp launcher, `POST /agents/{name}/trigger` returns `500` with an RFC 7807 body) rather than silently passing an empty string as a secret.
- **Two concurrent HTTP triggers for the same agent** (API/webapp launcher): both spawn independently with no de-dup or locking — asymmetric with the schedule launcher's own-timer overlap-skip, and deliberately so (an HTTP caller retrying or two legitimate concurrent requests are a different situation from one timer double-firing on itself); document the asymmetry explicitly so it doesn't read as an inconsistency.
- **Streamed trigger response and a slow/verbose child process**: the response streams chunk-by-chunk rather than buffering the full output in memory, and captured output forwarded to the launcher's own logs is length-capped (mirroring `server/src/engine/sandbox.rs`'s captured-output handling) so one chatty agent invocation cannot exhaust memory or flood `docker logs` unboundedly.
- **The webapp launcher's introspection route must never leak secret values** — `GET /_launcher/agents`/`GET /_launcher/agents/{name}` include field/env-var *names* and the JSON Schema, never `${VAR}`-resolved values, matching `bae-config.toml`'s existing "secrets... never persisted, never echoed" posture.
- **No `/healthz` for the schedule launcher**: it has no HTTP surface at all, by design — liveness is process-level only (`docker ps`, container exit status); this is a deliberate contrast with the API/webapp launchers (which do have `/healthz`), not a gap to fill.
- **A user's own Dockerfile accidentally overrides `ENTRYPOINT`**: the launcher binary then never runs even though the image builds successfully. Base image comments (matching the root `Dockerfile`'s own comment-heavy style) must call this out prominently, and the example Dockerfiles in `examples/launchers/` must demonstrate the correct pattern (only `COPY`, never `ENTRYPOINT`/`CMD`).
- **`bae-launcher-webapp` image built with zero configured agents**: the home grid shows an explanatory empty state (per section D), not a blank page — matching the project's established empty-state convention.
- **`BAE_LAUNCHER_API_TOKEN` left unset in a network-reachable deployment**: the API/webapp launcher's trigger routes are wide open. This is the documented V1 default (matching the admin port's pre-work-item-0004 posture) — call it out loudly in both the base image's build output/logs and `docs/guides/harness-launchers.md`, with an explicit pointer to `aspec/architecture/security.md`'s "TLS-terminating reverse proxy, internal network only" guidance.

## Test Considerations:

- **Unit — config parsing, always against multi-agent fixtures.** Every fixture `bae-schedules.toml`/`bae-api.toml`/`bae-app.toml` used across this test plan has **at least two** `[[agents]]` entries by default (a single-agent fixture is only used for the one test that specifically targets single-agent behavior), so the common case under test is always the N-agent case, not the 1-agent case: valid multi-agent fixtures parse into the expected N-length struct; malformed TOML, an invalid cron expression, an invalid JSON Schema, and a duplicate `name` (across two entries in an otherwise-valid multi-agent fixture) each produce the documented fatal startup error (exit 2) and a message naming the offending agent/field.
- **Unit — `${ENV_VAR}` templating**: a set env var resolves correctly into the spawned command's env/args; an unset one produces the documented per-invocation failure (not empty-string substitution), for both `launcher-schedule` and `launcher-api`.
- **Unit — JSON Schema validation (`launcher-api`)**: a valid POST body passes through to templating; an invalid one returns `400` with an RFC 7807 body naming the failing schema path, without ever spawning the child process.
- **Unit — log-line attribution**: captured output from two agents' concurrently-running child processes, interleaved at the byte level, is forwarded with each line correctly prefixed by its own agent's `name` — no cross-attribution between agents' output.
- **Integration — schedule launcher, multi-agent**: register **two or more** jobs on independent short-interval test schedules (e.g. one firing every second, another every two seconds) against fake `command`s that each write to their own file/counter; assert each fires the expected number of times independently of the other (proving no cross-agent coordination/throttling), that a deliberately slow fake command causes the documented overlap-skip **for that agent only** while the other agent's schedule keeps firing on time, and that a hung agent doesn't delay or block the other's fires.
- **Integration — API launcher, multi-agent**: boot `baeapi` against a temp config with **two or more** agents (distinct fake `command`s, distinct `request_schema`s) and drive `POST /agents/{name}/trigger` for each independently, asserting: each route validates against its own agent's schema (a body valid for one agent's schema but not another's is rejected on the right route); streamed output arrives incrementally per request; concurrent triggers to two different agents' routes complete independently with no interference; `BAE_LAUNCHER_API_TOKEN` (when set) is enforced uniformly across every agent's route but not on `/healthz`/`/_launcher/agents`.
- **Integration — webapp launcher, multi-agent**: boot `baeapi` with `BAE_LAUNCHER_WEBAPP_STATIC_DIR` pointed at a built `launchers/webapp/web/dist`, config with **two or more** agents; assert `GET /_launcher/agents` returns all of them with the documented safe fields only (no `env`/`env_template` values), and the static SPA's `index.html` is served at `/` including for an unknown client-side route (SPA fallback).
- **Regression — hung-child isolation**: a deliberately hanging fake agent command does not block a schedule launcher's other configured timers, nor an API launcher's other routes/agents, from continuing to function.
- **Regression — single-agent configs still work**: exactly one `[[agents]]` entry is a valid, fully-supported degenerate case of the N-agent path for all three launchers — not a separate code path — verified once per launcher so multi-agent-focused testing above doesn't accidentally leave the N=1 case untested.
- **Regression — image smoke tests**: `make image-launcher-schedule`/`-api`/`-webapp` each build; a minimal example Dockerfile from `examples/launchers/*` extending each base image builds and, when run, produces the expected behavior (a scheduled echo fires; a curl trigger returns the expected streamed output; the webapp's `/healthz` and SPA index both respond).
- **E2E (webapp, browser-driven)**: the home grid renders **one card per configured agent from a fixture `bae-app.toml` with three or more agents**; opening each card's detail page shows its own chat view scoped to that agent; sending a free-form message and, separately, clicking a pre-defined prompt button, both trigger `/agents/{name}/trigger` for the *correct* agent and render the streamed response live, without cross-talk between two agents' chat views opened in separate tabs.
- **Unit — `launchers/core`, tested standalone**: `resolve_env_refs`, `spawn_and_stream`'s log-line prefixing/length-capping, and `validate_unique_names` each get their own unit tests directly in `launchers/core`'s own test suite (not only exercised indirectly through `launchers/schedule`/`launchers/api`), since it is an independently buildable/testable component per `COMPONENTS` (section E).
- All new component tests remain fully offline (`make test-launchers/core`, `make test-launchers/schedule`, `make test-launchers/api`, `make test-launchers/webapp`) — no real provider keys, no real network calls, matching every existing component's test posture (`aspec/devops/cicd.md`: "All tests are offline").
