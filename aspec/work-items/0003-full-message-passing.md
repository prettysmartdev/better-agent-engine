# Work Item: Feature

Title: full message passing
Issue: issuelink

## Summary:
- The server should pass all events for the given session to any agent running on that session. That said, the server should not only send user/assistant messages to connected clients, but all message types (other than the ones generated by the client itself). This means that if an MCP tool call is done server-side, the client harness is still notified and can display information to the user if it desires. 

- **Wire protocol, cemented explicitly: the client port is a hybrid of HTTP/REST and JSON-RPC 2.0 — not JSON-RPC-only.** `aspec/work-items/0002-session-and-auth.md` already said the client port exposes a JSON-RPC API, but its own Implementation Details section, the code that actually shipped (`server/src/api/client/*.rs`), and `docs/client-api.md` all describe a plain REST/JSON surface instead (`POST /api/v1/sessions`, `POST /api/v1/sessions/{id}/messages`, `GET /api/v1/sessions/{id}/events`, `DELETE /api/v1/sessions/{id}`, no JSON-RPC envelope anywhere) — the spec's stated intent was never actually built. This work item reconciles that, but not by making everything JSON-RPC: liveness (`GET /healthz`), server metadata (`GET /api/v1/meta`), and all session CRUD/management (`POST /api/v1/sessions`, `GET /api/v1/sessions/{id}/events`, `DELETE /api/v1/sessions/{id}`) remain plain HTTP/REST, exactly as implemented today. What changes is the turn/event loop: a single new per-session endpoint, `POST /api/v1/sessions/{id}/rpc`, is where a harness sends a message into a session and receives the resulting stream of events, and it speaks JSON-RPC 2.0 and only JSON-RPC 2.0. Calling its message-send method is not "one request in, one JSON object out" — the server streams a sequence of JSON-RPC objects back over that same request/connection, one JSON-RPC notification per event the turn generates, followed by a single terminal JSON-RPC response — so one connection/request carries many server messages, and the server and every client SDK must be built to consume a *stream* of JSON-RPC objects off that one endpoint rather than a single buffered JSON body. The same endpoint also carries two new streaming-only methods, `session.subscribe`/`session.unsubscribe`, for a second connection to observe a session's live feed without driving it. The **admin port** (`BAE_ADMIN_ADDR`) is untouched by any of this: it stays standard HTTP/REST exactly as implemented today. See "Wire protocol — client port is REST for management, JSON-RPC 2.0 for the session loop" under Implementation Details for the full endpoint/method tables and transport framing.

Also, wire up MCP servers properly. The server can either be started with a `bae-config.toml` which enumerates all of the MCP servers that should be made available. Profiles can then reference configured MCP servers to make them available to clients assigned to a given profile. The profile-to-mcp relationship is OPT-IN meaning that if a profile lists no available MCP servers, NONE are connected to any client that uses that profile. If a profile lists an MCP server, it is then configured to be included when connecting to the LLM provider. If a profile lists an MCP server that does not exist, an error should be logged by the server and that MCP server should be skipped when configuring sessions for clients using that profile. It is not a fatal issue. The error message should be logged each time a client using a profile with an invalid MCP server creates a session.

In the docs/ folder, create a full set of user-facing documents explaining how to use bae with guides, reference, and examples.

## User Stories

### User Story 1:
As a: Agent Developer

I want to:
see every event the server generates on my session's behalf while a turn runs — provider requests/responses, tool calls, and real MCP request/response traffic — not just the final assistant message

So I can:
surface live progress in my client harness (e.g. "calling MCP server `search`...") and debug what actually happened during a turn without polling and reassembling the log myself

### User Story 2:
As a: Platform Operator

I want to:
declare a fixed set of MCP servers once in `bae-config.toml` and then opt individual profiles into a subset of them by name

So I can:
give different client profiles access to different tools (e.g. a "read-only" profile with no filesystem MCP server, a "power-user" profile with several) without duplicating server connection details per profile, and without a typo'd server name silently taking down session creation for a whole profile

### User Story 3:
As a: New bae User

I want to:
open `docs/` and find a guided path (guides → reference → examples) for running the server, wiring up MCP servers, and building a client harness

So I can:
get from zero to a working agent without reading the Rust source

## Implementation Details:

### Wire protocol — client port is REST for management, JSON-RPC 2.0 for the session loop

- **This is a correction, not a tightening of wording.** `aspec/work-items/0002-session-and-auth.md` (Summary, lines 9/13/15) says the client port serves "JSON-RPC over HTTP/2," but that same work item's Implementation Details section then specifies — and `server/src/api/client/mod.rs` + `sessions.rs` + `docs/client-api.md` actually implement — a plain REST/JSON `/api/v1/sessions*` surface with no JSON-RPC envelope (`jsonrpc`/`method`/`id`) at all. `aspec/architecture/apis.md` ("Convention: rest / Protocol: http") and `aspec/architecture/design.md` (Principle 2: "the wire protocol (`/api/v1 REST`) is small and stable") both currently describe that shipped REST reality, not the original JSON-RPC intent. From this work item forward, the client port is a deliberate **hybrid**: everything that is CRUD/management — opening a session, reading its persisted events, closing it, reading server metadata — stays plain HTTP/REST exactly as implemented today; only the live, session-bound message/event loop moves to JSON-RPC 2.0, on one new per-session endpoint. This is a breaking change to the message-send call in every existing client SDK, which is acceptable per the project's alpha status (README: "Status: alpha... APIs and SDKs will likely change") — the other endpoints are unaffected.
- **REST stays REST.** `GET /healthz`, `GET /api/v1/meta`, `POST /api/v1/sessions`, `GET /api/v1/sessions/{id}/events`, and `DELETE /api/v1/sessions/{id}` are unchanged from today's implementation — same routes, same request/response bodies, same RFC 7807 error shape documented in `docs/client-api.md`. This work item does not touch any of them; `meta` and session management are not JSON-RPC methods.
- **One new endpoint carries the session loop.** `POST /api/v1/sessions/{id}/rpc` is the single JSON-RPC 2.0 endpoint on the client port, scoped to exactly the harness-to-session interaction that needs streaming: sending a message into a turn and observing the resulting events, live. It replaces today's `POST /api/v1/sessions/{id}/messages` (single buffered response) with the JSON-RPC method `session.sendMessage`, and adds two new methods with no REST equivalent today, `session.subscribe`/`session.unsubscribe`. Nothing else is dispatched through this endpoint — it is not a general-purpose RPC surface, it is the session loop's endpoint, and its path carries the session id exactly like the other per-session REST routes.
- **`GET /healthz` stays exactly as implemented today** — it was never in scope for any of this: plain HTTP, unauthenticated, on the client port.
- **Auth is unchanged in shape.** The existing `Authorization: Bearer <key>` HTTP header applies to `POST /api/v1/sessions/{id}/rpc` exactly as it does to every other session-scoped route today: a session key is required and must match the `{id}` path segment (today's "session key bound to its session" check, unchanged). Because the session id is already a path segment — the same convention as `/api/v1/sessions/{id}/events` and `/api/v1/sessions/{id}/messages` today — JSON-RPC `params` do not need to carry `session_id` redundantly.
- **Framing for responses applies only to `POST /api/v1/sessions/{id}/rpc`.** Its HTTP response body is a stream of newline-delimited JSON-RPC 2.0 objects (`Content-Type: application/x-ndjson`), never a single buffered JSON document — even for methods that only ever emit one object — so a harness's JSON-RPC transport layer parses this one endpoint's responses uniformly regardless of method. The HTTP response is not "done" until the terminal object (a JSON-RPC response whose `id` matches the request, or a JSON-RPC error object) has been written and the body closed. Every other client-port endpoint (`meta`, session open/getEvents/close) keeps returning a single buffered JSON body, exactly as today.
- **Notifications vs. the terminal response** use JSON-RPC 2.0's own envelope rules, not a bae-specific convention, and only apply within the `/rpc` endpoint's stream: an object with no `id` member is a notification (informational, no reply expected); an object carrying the request's original `id` is the terminal response. A harness's JSON-RPC transport layer only needs to branch on "does this object have an `id`" once, and that same branch serves both `session.sendMessage`'s inline event notifications and `session.subscribe`'s long-lived notification stream (see "Event broadcasting" below).
- **Endpoint table** (supersedes the relevant rows of `docs/client-api.md` and of 0002's Implementation Details):

  | Endpoint | Protocol | Status | Auth |
  |---|---|---|---|
  | `GET /healthz` | REST | unchanged | none |
  | `GET /api/v1/meta` | REST | unchanged | none |
  | `POST /api/v1/sessions` | REST | unchanged | client key |
  | `GET /api/v1/sessions/{id}/events` | REST | unchanged | session key |
  | `DELETE /api/v1/sessions/{id}` | REST | unchanged | session key |
  | `POST /api/v1/sessions/{id}/rpc` | JSON-RPC 2.0 | **new** — replaces `POST /api/v1/sessions/{id}/messages`, carries the methods below | session key |

  **JSON-RPC methods dispatched on `POST /api/v1/sessions/{id}/rpc`:**

  | Method | Replaces | `params` | Streams notifications? |
  |---|---|---|---|
  | `session.sendMessage` | `POST /api/v1/sessions/{id}/messages` | `{message}` | Yes — see "Event broadcasting." |
  | `session.subscribe` | *(new — see "Event broadcasting")* | `{since_event_id?}` | Yes, indefinitely — no terminal response while active. |
  | `session.unsubscribe` | *(new)* | `{}` | Ends an open `session.subscribe` call cleanly. |

- **JSON-RPC batch requests (arrays of request objects) are explicitly unsupported** on `POST /api/v1/sessions/{id}/rpc` — reject with a `-32600 Invalid Request` error rather than silently accepting one. Batch responses in the JSON-RPC 2.0 spec are returned as a single array once every sub-request completes, which is incompatible with streaming a method like `session.sendMessage` incrementally; rather than special-case batching around that, this work item disallows it outright and documents that explicitly so it isn't discovered by trial and error.
- **Malformed or non-JSON-RPC bodies posted to `POST /api/v1/sessions/{id}/rpc`** get a proper JSON-RPC error object (`-32700 Parse error` for invalid JSON, `-32600 Invalid Request` for a well-formed JSON value that isn't a valid request object, `-32601 Method not found`, `-32602 Invalid params`). This is scoped to that one endpoint — every other client-port endpoint (`meta`, session open/getEvents/close) keeps returning the RFC 7807 problem-detail body `docs/client-api.md` documents today, unchanged, and so does the admin port.
- **Update `aspec/architecture/apis.md`** ("Convention: rest / Protocol: http", and the closing line "long-running run output streamed via SSE") and **`aspec/architecture/design.md`** (Principle 2's "the wire protocol (`/api/v1 REST`) is small and stable" and the mermaid diagram's `REST API /api/v1 (axum)` label) so these architecture-level aspec documents stop contradicting this work item: both should describe the client port as REST for management plus one JSON-RPC 2.0 endpoint for the session loop, and the admin port as REST/HTTP throughout (no SSE anywhere on either port).

### Event broadcasting — session-scoped fan-out over JSON-RPC

- Today (pre-this-work-item), `POST /api/v1/sessions/{id}/messages` already returns every `EventRecord` logged during that one synchronous turn (`server/src/api/client/sessions.rs`, `run_turn` in `server/src/engine/session.rs`) — but only to the connection that made the request, only as one buffered JSON body, and only after the whole turn finishes. There is no mechanism for events to reach a client that isn't the one who triggered the turn, and no incremental/live delivery while a turn (especially one with several MCP round-trips) is still in progress. This work item fixes both problems using the JSON-RPC streaming model from "Wire protocol" above — **there is no separate SSE endpoint**; live delivery is JSON-RPC notifications on the one `POST /api/v1/sessions/{id}/rpc` endpoint.
- Add an in-process broadcast registry: `HashMap<SessionId, tokio::sync::broadcast::Sender<EventRecord>>` guarded by a `Mutex` (or `DashMap` if we want to add that dependency), owned by `AppState`. `tokio`'s `sync` feature is already enabled in `server/Cargo.toml`, so `broadcast` needs no new dependency.
- Introduce a single choke point for persisting events — e.g. `Store::log_event` or a thin wrapper called everywhere `session.rs`/`sessions.rs` currently insert an `EventRecord` — that, after the DB insert succeeds, also publishes the record on that session's broadcast channel (lazily created on first subscribe; `send()` on a channel with no receivers is a harmless no-op, ignore its `Err`).
- **`session.sendMessage` streams its own turn's events.** Once dispatched (via `POST /api/v1/sessions/{id}/rpc`, per the endpoint/method tables above), the handler writes a `{"jsonrpc":"2.0","method":"session.event","params":<EventRecord>}` notification for every `EventRecord` the turn produces, in order, as it happens — not just a buffered list — followed by exactly one terminal JSON-RPC response object (carrying the original request's `id`) whose `result` is the `{message}` shape returned today, written the moment the turn completes (or a JSON-RPC error object on unrecoverable provider failure — see edge cases). Concretely, this is what "the messages call streams back multiple JSON-RPC objects in a single response" means: N `session.event` notifications, then 1 terminal response, all on the HTTP response body opened by that one call — the driving client gets live progress on its own connection without a second subscription.
- A client that doesn't care about live progress can simply discard notification objects and await only the terminal response — the `{message}` result shape is unchanged from today's `POST /messages` body, so nothing is lost for simple callers even though the transport now always streams.
- **`session.subscribe` is how a second, non-driving connection gets the same live feed** (the "observer" case — e.g. a dashboard watching a session it isn't driving). Called on the same `POST /api/v1/sessions/{id}/rpc` endpoint, authenticated with that session's key exactly like `session.sendMessage`; `params: {since_event_id?}` — no `session_id`/`session_key` in `params` since the id is already the path segment and the key is already the `Authorization` header. On call, the server first replays persisted events after `since_event_id` (if given) as `session.event` notifications via the existing `session.getEvents`-style query, then seamlessly switches to the live broadcast subscription and keeps emitting `session.event` notifications indefinitely. This call has **no terminal response** while the subscription is active — it ends only when the client disconnects or calls `session.unsubscribe`.
- **Filtering rule (per the summary), shared by both `session.sendMessage`'s inline notifications and `session.subscribe`:** forward every `EventType` *except* ones the client itself generated in the request that produced them: `client.message.send`, and any `tool.result` events whose payload indicates client-side dispatch (`dispatch: "client"`, submitted by the very connection now listening) are not echoed back. Everything else — `provider.request`, `provider.response`, `tool.call`, MCP-dispatched `tool.result`, `mcp.request`, `mcp.response`, `server.message.send`, and all `session.*` events — is forwarded. Implement the filter as a predicate function (`server/src/engine/broadcast.rs` or similar) shared by the publish call site so the rule lives in exactly one place.
- Document that a client should track the last event ID it saw (from either `sendMessage`'s notifications or `subscribe`'s) and always reconnect/resubscribe with `since_event_id` to avoid gaps.
- On `tokio::sync::broadcast::error::RecvError::Lagged`, close the stream with a distinguishable JSON-RPC error object (e.g. `{"jsonrpc":"2.0","error":{"code":-32000,"message":"lagged; reconnect with since_event_id"}}`) rather than silently dropping events, so the client knows to reconnect/resubscribe with `since_event_id` and reconcile via `GET /api/v1/sessions/{id}/events` rather than assuming it saw everything. Shared by both `sendMessage` and `subscribe`.
- On session close (`DELETE /api/v1/sessions/{id}`), drop the session's broadcast sender from the registry so idle entries don't accumulate.

### MCP server registry — `bae-config.toml`

- Add a `toml = "0.8"` dependency to `server/Cargo.toml` (confirmed absent today).
- New optional startup input, settable **either** way: the `BAE_CONFIG` env var **or** a `--config <path>` CLI flag on `baesrv` (`baesrv --config path/to/bae-config.toml`, or `baesrv serve --config ...` explicitly — `serve` is the default subcommand either way), giving a path to a TOML file. When both are set, `--config` wins — this is the first CLI flag `cli.rs` gets (today it only does ad hoc subcommand matching, no flag parsing at all), and it establishes the flag-beats-env-var precedence `aspec/uxui/cli.md`'s Configuration section already commits to ("If a config file is ever added it will be explicitly opted into via `--config <path>`, with env/flags still taking precedence."). Default: neither set — matches the existing "everything is opt-in via env vars" pattern in `server/src/config.rs`, whose doc comment currently says "there is no config file" and will need updating. Absence of both (or of the file the resolved path points to) is not an error — the server starts with an empty MCP server registry, meaning no profile can resolve any MCP server (fully consistent with "opt-in" — nothing is available unless explicitly configured).
- **The file must be extensible beyond MCP from day one.** MCP servers are *not* top-level entries in `bae-config.toml` — they live under a top-level `mcp` key, so future top-level keys (e.g. a later `[logging]`, `[providers]`, or `[limits]` section) can be added without colliding with or restructuring the MCP section. Shape:
  ```toml
  [mcp]

  [[mcp.servers]]
  name = "filesystem"
  transport = "stdio"
  command = "npx"
  args = ["-y", "@modelcontextprotocol/server-filesystem", "/data"]

  [[mcp.servers]]
  name = "search"
  transport = "sse"
  url = "https://mcp.example.com/sse"
  headers = { Authorization = "Bearer ${SEARCH_MCP_TOKEN}" }
  ```
  Model this in Rust as `struct BaeConfig { mcp: Option<McpConfig> }` / `struct McpConfig { servers: Vec<McpServerConfig> }` (top-level struct deliberately has room for sibling fields later) rather than a flat `Vec<McpServerConfig>` parsed directly off the document root. A `bae-config.toml` with no `[mcp]` table at all is valid and yields an empty registry, same as a missing file.
  `name` must be unique within `mcp.servers` — **fail startup** (this is an operator config error, distinct from the per-profile "skip and log" case below) if two entries share a name, or if `transport` is anything other than the supported set. Reuse the existing `${ENV_VAR}` substitution helper already used for provider `auth_token` resolution in `session.rs` (extract it to a shared util if it isn't already reusable) for `headers`/auth values — resolve immediately before connecting, never persist the resolved value, matching the existing provider-config convention.
- Build a registry `HashMap<String, McpServerConfig>` on `AppState` at startup from `config.mcp.map(|m| m.servers).unwrap_or_default()`.
- Add a read-only admin endpoint, `GET /admin/v1/mcp-servers`, listing configured server names and transport types (no secrets) — useful for an operator to confirm what a running server actually has available, since the registry is config-file-driven rather than DB-driven like profiles/keys. This lives on the admin port, so it's plain HTTP/REST like every other admin endpoint — the client-port's REST/JSON-RPC split above is specific to `BAE_ADDR` and doesn't apply here.
- **Example config files**: add an `examples/bae-config/` directory at the repo root with a few ready-to-run example TOML files built from real, commonly-used MCP servers (not synthetic placeholders) so an operator can copy one and go:
  - `examples/bae-config/filesystem.toml` — a single `[[mcp.servers]]` entry running the official `@modelcontextprotocol/server-filesystem` over stdio via `npx`.
  - `examples/bae-config/fetch.toml` — a single entry running the official fetch/web-access MCP server (stdio, via `uvx mcp-server-fetch` or equivalent) for outbound HTTP fetch tool access.
  - `examples/bae-config/multi-server.toml` — combines filesystem + fetch (+ one more, e.g. the official git server) under one `[mcp]` table, demonstrating multiple `[[mcp.servers]]` entries and mixed transports (stdio + a placeholder `sse` entry).
  Each example file gets a short comment header naming the runtime prerequisite (Node.js/`npx` vs Python/`uvx`) and the upstream package it runs. These files are referenced directly from `docs/guides/mcp-servers.md` (below) rather than duplicated inline in the doc, so the doc and the runnable example never drift apart.

### Config module structure (config.rs)

- **Remove `BAE_TLS_ENABLED`/`Config::tls_enabled` entirely** — from `server/src/config.rs` (field, parsing, `ConfigError::InvalidBool` and the now-unused `parse_bool` helper, and its tests) and from every doc that lists it (`server/README.md`, `docs/quickstart.md`, `aspec/devops/operations.md`, and the stale mention in `aspec/work-items/0002-session-and-auth.md`). It never did anything — the container has always spoken plain HTTP internally regardless of its value — so keeping it around as a documented, validated env var is actively misleading about what flipping it does. The actual architectural fact (TLS terminates upstream) stays documented; only the fake toggle goes.
- `server/src/config.rs` currently loads only `BAE_*` env vars into a flat `Config` struct, and `server/src/cli.rs::run` currently does bare subcommand string-matching on `std::env::args()` with no flag parsing anywhere. Add both a CLI flag and TOML-file loading as distinct steps: `cli.rs` gains minimal ad hoc parsing (matching its existing style — no need to pull in a flag-parsing crate for a single flag) for `--config <path>`, passed down to `load_config`/`run_serve`/`run_migrate`; `Config::from_env()` continues to resolve `BAE_*` env vars as today; a new `BaeConfigFile::load(path: Option<&Path>)` (or similar) separately loads and parses the TOML file at whichever path resulted from `--config` (if given) or else `BAE_CONFIG` (if set). Keep these three loading concerns distinct in code (CLI-flag parsing, env-driven server config, file-driven MCP registry) since they have different failure semantics: env var/flag config errors are usage errors (exit code 2, per `aspec/uxui/cli.md`) at every startup, whereas a missing config file (via either `--config` or `BAE_CONFIG`) is not an error at all.

### Profile → MCP wiring (opt-in, non-fatal on typo)

- `ProfileRecord.mcp_servers` (`server/src/store/profiles.rs`) changes meaning from an opaque JSON blob to a JSON array of **strings** — names that must match entries in the `bae-config.toml` registry. The admin API (`server/src/api/admin/profiles.rs`) already validates `mcp_servers` is an array (`require_array`); tighten that to also require every element be a string. This is a shape change to an alpha API — no DB migration needed (still a `TEXT` column, same `ProfileRecord.mcp_servers: Value`), but call it out as a breaking change to any existing profile data in the docs (per README: "Status: alpha... APIs and SDKs will likely change").
- Resolution happens **at session-creation time** (`POST /api/v1/sessions`, `server/src/api/client/sessions.rs::create` handler — unchanged REST route), not at profile admin-create time — the registry is server-config-driven and may differ from what existed when the profile was written. For each name in the profile's `mcp_servers`:
  - If found in the registry: mark it as active for this session (see connection lifecycle below).
  - If **not** found: `tracing::error!(profile_id, profile_name, mcp_server_name, session_id, "configured MCP server not found in bae-config.toml; skipping")` and continue — this must **not** fail session creation, and must log **every time** a session is created against that profile (not once/deduplicated), exactly as the summary specifies.
- The set of resolved MCP servers for the session determines which tools get merged into the provider request (see below) and which tool names route to the `"mcp"` dispatch path in `session.rs`'s existing `has_client_tool` / dispatch-tagging logic.

### Real MCP client integration (replacing the stub in `engine/session.rs`)

- `engine/session.rs` today has a fully-formed **stub** MCP path (lines ~274-311 per current code): it logs `McpRequest`/`McpResponse` with `{"status": "stub", ...}` payloads and fabricates a canned text tool result, without ever leaving the process. This work item replaces that with a real MCP client, keeping the same `EventType::McpRequest` / `EventType::McpResponse` / `EventType::ToolResult` event slots (no new `EventType` variants needed — the 12-variant closed enum stays as-is; only payload contents change from stub shape to real shape).
- Add a new `server/src/engine/mcp.rs` module (mirroring the existing shape of `server/src/engine/provider.rs`) responsible for: spawning/connecting to a configured MCP server (stdio: spawn subprocess per session for isolation; sse/http: open a connection per session), performing the MCP `initialize`/`tools/list` handshake, and dispatching `tools/call`. Evaluate an existing Rust MCP client crate (e.g. `rmcp`, the official Rust MCP SDK) rather than hand-rolling the protocol; if none fits, hand-roll only the JSON-RPC subset actually needed (`initialize`, `tools/list`, `tools/call`).
- At session creation, for each resolved MCP server: connect, call `tools/list`, and merge the returned tool definitions into the tool list sent to the provider alongside the client-declared tools — build and retain a `tool_name -> server_name` map for routing tool_use blocks during the turn.
- In `run_turn`'s existing MCP dispatch branch, replace the stub block with: log `McpRequest` with the real `tools/call` request payload, invoke it via the `mcp.rs` client for the owning server, log `McpResponse` with the real response (or an error payload on failure — see edge cases), and build the `ToolResult` from the real response content instead of the canned string.
- Include `server_name` in the `tool.call` / `tool.result` / `mcp.request` / `mcp.response` payloads (these are freeform `serde_json::Value` payloads already — no schema change) so the event log and any downstream `session.event` notification consumer can show which configured MCP server handled a given tool call.
- On session close, terminate any spawned stdio subprocesses and close any open MCP connections for that session — pair this cleanup with the broadcast-channel cleanup mentioned above.
- Note the tension with `aspec/architecture/design.md`'s "Principle 2" (server is out of scope for "tool implementations... those live in client harnesses"): MCP dispatch is a deliberate, scoped exception, since an MCP server is an external process/service the *operator* configures, not application tool logic living in a client harness. Worth a short callout in `aspec/architecture/design.md` alongside this work, since a future reader will otherwise see it as a contradiction.

## Edge Case Considerations:

- **Neither `--config` nor `BAE_CONFIG` set, or the resolved file missing**: not an error; server starts with an empty MCP registry. Any profile listing MCP servers behaves exactly like the "server not found" case below for every entry, every session.
- **Duplicate `name` within `bae-config.toml`, or unsupported `transport` value**: fail server startup with a clear message — this is an operator authoring error caught once, not a per-session runtime concern.
- **Profile references an MCP server name not in the registry**: log an error (as specified) and skip only that server; session creation still succeeds with whatever subset of requested servers did resolve (including zero). The error must be logged on **every** session creation against that profile, not just once — do not deduplicate/cache "already warned about this name."
- **MCP server resolves by name but fails to connect** (stdio binary missing/non-executable, sse endpoint unreachable, referenced env var for an auth header unset): treat the same as "not found" for session-creation purposes — log and skip, non-fatal — rather than failing the whole session, since a client should still be able to use the profile's other tools and the LLM provider even if one MCP server is down.
- **MCP subprocess/connection dies mid-turn** (after successful connect, before/during a `tools/call`): log `McpResponse` with an error payload, synthesize an error-shaped `ToolResult` (so the model can see the tool failed and adjust) rather than terminating the whole turn; do not attempt to reconnect mid-turn.
- **Concurrent sessions using the same configured stdio MCP server**: default to one subprocess per session for isolation; note the resource cost explicitly and flag process-pooling/sharing as a future optimization, not required here.
- **Session close / crash cleanup**: spawned MCP subprocesses and open MCP connections must be terminated when a session closes; the session's broadcast channel entry must be removed from the registry so it doesn't accumulate indefinitely across many short-lived sessions.
- **A `session.subscribe` (or `session.sendMessage`) client is slow or has disconnected without the server noticing yet**: use a bounded broadcast channel; on `Lagged`, close the stream with an explicit JSON-RPC error object rather than silently skipping — instruct the client (in docs) to reconnect/resubscribe with `since_event_id` and treat the notification stream as at-least-once, not gap-free, without that reconciliation.
- **No active subscriber on a session's broadcast channel**: publishing must be a harmless no-op (ignore the `Err` from `Sender::send` when there are zero receivers) — most turns will only have the driving `session.sendMessage` call itself consuming notifications, with no separate `session.subscribe` connection.
- **Multiple simultaneous connections to one session** (e.g. one connection driving `session.sendMessage`, another only running `session.subscribe`): both are authenticated with the same session key; this is allowed and is in fact the main new use case (a second "observer" agent/dashboard watching a session it isn't driving).
- **Reconnect ordering**: `since_event_id` replay-then-live-subscribe must not produce gaps or (ideally) duplicates at the seam — document the small window where a duplicate is possible rather than silently promising perfect exactly-once delivery.
- **A JSON-RPC batch request (array of request objects) is posted to `POST /api/v1/sessions/{id}/rpc`**: reject with a single `-32600 Invalid Request` error rather than attempting partial batch semantics — batching is explicitly unsupported, see "Wire protocol" above.
- **Malformed JSON, a JSON value that isn't a valid JSON-RPC request object, an unknown `method`, or invalid `params`, posted to `POST /api/v1/sessions/{id}/rpc`**: respond with the corresponding JSON-RPC error object (`-32700`/`-32600`/`-32601`/`-32602`) rather than the RFC 7807 problem-detail body used everywhere else on the client port and on the admin port — this one endpoint has a genuinely different error envelope from the rest of the API surface, and that must be documented, not left implicit.
- **All providers fail during `session.sendMessage`**: previously surfaced as an HTTP `502` whose body was still `{message, events}`; now surfaces as the terminal JSON-RPC response carrying the same generic "provider unavailable" assistant turn in `result.message`, with the session moved to `error` state exactly as before — HTTP status stops being a meaningful signal for provider failures on `POST /api/v1/sessions/{id}/rpc` specifically (every other client-port endpoint keeps using HTTP status normally), so `docs/client-api.md`'s "502 special case" callout is replaced with this JSON-RPC-shaped equivalent, scoped to that one endpoint.
- **Existing profiles with the old opaque `mcp_servers` JSON blobs** (pre-this-work-item shape): calling this out as a breaking, undocumented-migration change is acceptable per the project's alpha status, but must be stated plainly in the docs/changelog so an operator isn't surprised when their old profile's `mcp_servers` entries are silently ignored (non-string array elements) rather than erroring loudly — decide and document one of: reject non-string entries at admin-update time going forward, or treat them like any other "not found by name" case.

## Test Considerations:
