# Work Item: Feature

Title: max webapp
Issue: issuelink

## Summary:
- This work item adds a new sub-codebase called MAX (multi-agent executor) within BAE.

MAX is purely a webapp interface for administering and interacting with BAE. It is built into a SECOND dev and production container image (i.e. there will be one with ONLY baesrv and baectl, and a SECOND with baesrv, baectl, AND max). If the end-user chooses to get the 'bae-max' container variant, it will include the webapp.

The bae-max container image must automatically start BOTH the baesrv server AND the max webapp server, which will be a Vite + React webapp with Node.JS backend. It is capable of accessing BOTH the admin port (via the auto-generated credentials, identically to baectl) AND the client port (but ONLY as an observer, for now.) 

The max Node.JS backend understands how to interact with both ports made available by baesrv and presents web UIs for managing profiles and keys. It also has the ability to view currently running sessions, create session keys on the fly, and join any session as an observer, with a live websocket streaming events to the web app client and rendering them in a user-friendly graph. The graph should be interactive, showing full event details when an event node is clicked.

The app as a whole should be a simple dashboard style with 'max' text in the top left, and top-bar tabs for 'Keys', 'Profiles', and 'Sessions' (for the live session viewer).

The entire app MUST be mobile and tablet friendly.

## User Stories

### User Story 1:
As a: Platform Operator

I want to:
open a web dashboard — instead of `docker exec`-ing `baectl` or hand-assembling admin-API curl calls — to create/list/revoke profiles and client keys, from a desktop, tablet, or phone browser

So I can:
administer a running `bae-max` container without a terminal, a Rust toolchain, or memorizing the admin API's JSON shapes, the same zero-configuration way `baectl` already reaches the admin port from inside the container

### User Story 2:
As an: Agent Developer / Support Engineer

I want to:
open the Sessions tab, pick any session (running or historical), and watch its events render live as an interactive graph — client turns, provider requests/responses, tool calls, MCP exchanges, join/driver events — with a click on any node showing that event's full JSON payload

So I can:
debug or audit an in-flight or past agent run without writing my own observer harness, reconstructing a `session.subscribe` client, or grepping raw event JSON

### User Story 3:
As a: Platform Operator away from my desk

I want to:
use every part of the dashboard (Keys, Profiles, Sessions, the event graph) on a phone or tablet browser exactly as well as on desktop

So I can:
mint a key, check a stuck session, or triage an incident from whatever device is in front of me, not just a full workstation

## Implementation Details:

This work item has five mostly-independent pieces: (A) a new `max/` sub-codebase (Vite + React frontend + Node.js backend) bundled into a second `bae-max` container image variant; (B) two new **read-only** admin-port endpoints so MAX can list/inspect sessions without ever holding a session key; (C) a client-port bridge — per-profile "observer" client keys, `POST /api/v1/sessions/{id}/join`, and `session.subscribe` — that turns the existing NDJSON live-event stream into a genuine browser WebSocket; (D) MAX's own login/auth layer, required because this is the **first time** anything with effectively-admin capability is exposed off-loopback; (E) the dashboard itself (Keys/Profiles/Sessions tabs, the interactive event graph, mobile/tablet responsiveness).

**Dependency on work item 0004 (`baectl` CLI).** The summary's "via the auto-generated credentials, identically to baectl" assumes the bootstrap-admin-key-file mechanism designed in `aspec/work-items/0004-baectl-cli.md` — today implemented on branch `feature/0004-baectl-cli` (`server/src/admin_auth.rs`, `BAE_ADMIN_KEY_FILE` default `/var/lib/bae/admin-key.pem`) but **not yet merged into `main`**: as of this writing `main`'s `server/src/api/admin/mod.rs` still says "there is no auth on this port initially because it is localhost-only," and `baectl/` does not exist on `main` at all. This work item is written assuming 0004 lands first (or in the same stack); MAX's admin-port auth reuses `BAE_ADMIN_KEY_FILE`/`BAE_ADMIN_KEY_HASH_FILE` verbatim rather than inventing a second bootstrap mechanism. If 0004 has not landed when this is implemented, treat that as a blocking prerequisite, not something to route around.

### A. New component: `max/`

- Two sub-packages under `max/`, mirroring how `baectl/` stands alone from `client-rust/` (own manifests, no shared workspace with the Rust components):
  - `max/web/` — the Vite + React frontend (`npm create vite@latest -- --template react-ts` scaffold). Builds to a static `dist/` directory.
  - `max/server/` — the Node.js backend (Express + `ws`) that serves `web/dist` as static files, exposes MAX's own thin REST/WS surface to the browser, and holds every credential (admin key, per-profile observer keys, MAX's own login secret) — none of which the browser ever sees directly.
- Package names `@prettysmartdev/bae-max-web` / `@prettysmartdev/bae-max-server`, `"private": true` — like `baectl`, MAX ships only inside the `bae-max` image and has no independent npm publish story (see Codebase Integration's `cicd.md` note).
- `max/Makefile` exposes the standard `build`/`test`/`lint`/`fmt`/`clean` verbs (mirroring every existing component's Makefile — `client-typescript/package.json`'s `build`/`test`/`lint`/`clean` scripts are the closest analog), each verb running both `web/` and `server/`'s own npm scripts. Add `max` to the root `Makefile`'s `COMPONENTS` list (`server client-rust client-typescript client-python` → `... max`, alongside `baectl` once 0004 lands) so `make build`/`test`/`lint`/`fmt`/`clean` and the per-component `<verb>-max` targets work identically to every other component.
- Node 22 LTS toolchain in `Dockerfile.dev`, matching the version already installed there for `client-typescript` (nodesource `setup_22.x`) — no new dev-image toolchain needed, `max` reuses it.
- **Does not depend on `client-typescript` (`@prettysmartdev/bae-ts`) for the join+subscribe flow.** `Harness.connect()`/`Harness.join()` (`client-typescript/src/harness.ts:80-125`) unconditionally call `registerDriver` before returning a `Session` — by design, since every harness user is assumed to be a driver. Using it from `max/server` would silently register MAX as a **driver** on every session it merely observes, contradicting "ONLY as an observer, for now" and polluting the driver registry/event log with a phantom participant. Instead `max/server` implements its own minimal, purpose-built client-port client (a small `fetch`-based wrapper calling `POST /api/v1/sessions/{id}/join` and driving the NDJSON `session.subscribe` stream directly, never `session.registerDriver`/`session.sendMessage`) — the same scope-boundary reasoning `aspec/work-items/0004-baectl-cli.md` already gives for why `baectl` doesn't depend on `client-rust`.

### B. New read-only admin-port endpoints: session visibility without a session key

Today there is **no way to list sessions at all** — the admin port (`server/src/api/admin/mod.rs`) only exposes `profiles`, `keys`, `mcp-servers`, `providers`; session data lives entirely behind client-port, session-key-authenticated routes (`GET /api/v1/sessions/{id}/events`, `GET /api/v1/sessions/{id}/participants`, both in `server/src/api/client/sessions.rs`). Requiring MAX to hold a session key just to populate a session list (or to read the history of a `closed`/`error` session it was never connected to while open — `join` rejects with `409 session_closed` on a terminal session, so there is no client-port path to a historical session's events at all) is both impossible for terminal sessions and unnecessarily roundabout for the "browse everything" list view. This work item adds two new **admin-port, read-only** routes, mirroring the existing `mcp.rs`/`providers.rs` "config/state visibility without secrets" pattern:

- `GET /admin/v1/sessions` — cursor-paginated (same `PageQuery`/`next_cursor` convention as `profiles`/`keys`), optional `?state=open|closed|error` filter. New store function `sessions::list_sessions(conn, after: i64, limit: i64, state: Option<&str>) -> rusqlite::Result<(Vec<(i64, SessionRecord)>, bool)>` in `server/src/store/sessions.rs`, added next to `create_session`/`get_session` (`sessions.rs:82`, `:142`), reusing `SESSION_COLS`/`row_to_session` (`sessions.rs:56-74`). View: `{id, profile_id, state, client_version, created_at, closed_at}` — omits `client_tools` (not secret, but noisy JSON not needed for a list view).
- `GET /admin/v1/sessions/{id}/events` — the same event history `GET /api/v1/sessions/{id}/events` already serves (reuses `sessions::list_events` and `event_view`'s exact shape, `client/sessions.rs:625-654`), but admin-authenticated instead of session-key-authenticated — the only way to read a **closed/errored** session's full history without ever having held a session key for it. `404` if the session id doesn't exist.
- New admin handler module `server/src/api/admin/sessions.rs` (mirroring `admin/mcp.rs`'s/`admin/providers.rs`'s doc-comment style), registered in `admin/mod.rs`'s router next to the existing routes, and added to that file's top doc-comment endpoint list (`mod.rs:12-16`).
- These two routes are the **only** way MAX gets a session list or historical-session detail; MAX never queries SQLite directly (per `aspec/uxui/interface.md`'s "the UI is a pure API client" mandate — everything MAX can do must already be possible via the documented API).

### C. Client-port bridge: per-profile observer keys, `join`, and `session.subscribe`

For a session currently `open`, list/history alone isn't enough — the summary asks for a **live** tail. There is no server-side push mechanism on the admin port (by design — it's plain REST, per `docs/reference/wire-protocol.md`'s "The admin port ... is REST/HTTP throughout"), so the only way to get live events is the existing client-port machinery: a **client key** (role `client`) resolving to the session's profile, `POST /api/v1/sessions/{id}/join` to mint a session key, then `session.subscribe` over `POST /api/v1/sessions/{id}/rpc` (NDJSON). MAX therefore needs, and the summary explicitly grants, client-port access — "but ONLY as an observer, for now."

- **Per-profile observer client keys, lazily provisioned and persisted.** `join` enforces `client_key.profile_id == session.profile_id` (`403 profile_mismatch`, `client/sessions.rs:309-314`), so MAX cannot use one key for every session — it needs one client key **per profile**. On first attempt to observe any session under a given `profile_id`, `max/server` calls `POST /admin/v1/keys` with `name: "max-observer-<profile_id>"` and persists the plaintext (shown only once, per `admin/keys.rs:39-79`'s "the plaintext is returned exactly once in this response and never again") to a new file on the shared data volume, `/var/lib/bae/max-observer-keys.json` (default; overridable via `BAE_MAX_OBSERVER_KEYS_FILE`), written `0600` — the same durable, restart-surviving, `docker exec`-avoidable pattern `BAE_ADMIN_KEY_FILE` already establishes for the admin bootstrap key. Format: `{"<profile_id>": {"key": "bae_...", "key_id": "key_..."}, ...}`. Subsequent observe requests for the same profile read this file instead of minting a new key.
- **Session-key caching, in-memory only.** Once `max/server` joins a given `session_id` (minting a session key via the profile's observer client key), it caches `session_id -> session_key` in an in-process map for the life of the process — every subsequent browser tab watching that same session reuses the cached session key and the same upstream `session.subscribe` NDJSON stream (fanned out to N downstream browser WebSocket connections), rather than re-joining and logging a redundant `session.join` event per viewer. Lost on `max/server` restart (acceptable — a restart re-joins on next observe, exactly like `mcp_sessions`/`drivers`/`turn_gates` are documented as in-memory-only and rebuilt on the server side; see `aspec/work-items/0005-parallel-client-handling.md`'s Codebase Integration).
- **The bridge itself.** For a given session, `max/server`: (1) `GET /admin/v1/sessions/{id}/events` for the full persisted history (works even if the session later turns out to be closed by the time the browser asks); (2) if the session is `open`, `join` (or reuse the cached session key) and open the NDJSON `session.subscribe` stream (`{"jsonrpc":"2.0","id":1,"method":"session.subscribe","params":{"since_event_id": "<last id from step 1>"}}`), relying on the server's own replay→live dedup (`rpc.rs`'s `drive_subscribe`, replay/dedup logic at lines ~350-410 in the current `rpc.rs`) so no event is gapped or double-delivered; (3) forward each NDJSON line's `session.event` notification to every browser WebSocket subscribed to that session id (`wss://<max-host>/ws/sessions/{id}`), as newline-delimited JSON re-framed into a simple `{type: "event", event: {...}}` WS message.
- **Never registers as a driver, never sends a message.** `max/server`'s client-port usage is strictly `POST .../join` (REST) + `session.subscribe`/`session.unsubscribe` (JSON-RPC) — it must never call `session.registerDriver` or `session.sendMessage`, matching the summary's "ONLY as an observer, for now" and `rpc.rs`'s own framing of `subscribe` as "the observer registration act; no driver registration needed."
- **Lagged-subscriber recovery.** Per `docs/reference/wire-protocol.md`'s "Lagged subscriber" section, a `session.subscribe` stream that falls behind the 256-event broadcast buffer closes with a `-32000 "lagged; reconnect with since_event_id"` notification. `max/server` must catch this and immediately reconnect `session.subscribe` with `since_event_id` set to the last event id it forwarded — silently letting the stream die here would look to browser viewers like the session simply stopped producing events.

### D. MAX's own authentication (a genuinely new network-exposed surface)

Every existing admin capability (`baectl`, the raw admin API) is reachable only via `docker exec`/loopback — never off-host. `max/server`'s web port is the **first** thing in this project that exposes admin-equivalent capability (create/revoke keys and profiles, observe every session) to the operator's network. This is a required, not optional, piece of this work item — shipping it without its own auth would be a real security regression versus everything that exists today.

- New env var `BAE_MAX_PASSWORD` (explicit value, highest precedence) / `BAE_MAX_PASSWORD_FILE` (default `/var/lib/bae/max-password.pem`, read if `BAE_MAX_PASSWORD` unset) — mirroring the exact `--admin-token` > `--admin-key-file` > default-file-path precedence `baectl` already establishes (`aspec/work-items/0004-baectl-cli.md`'s "Auth / addressing" section). On first boot, if neither is set, `max/server` self-generates a random password and writes it to the default file path with `0600` permissions (same posture as `BAE_ADMIN_KEY_FILE`'s self-generate path), logging (never the value) that it did so.
- Unlike `--dangerously-disable-admin-auth`, there is **no** disable flag for MAX's own auth — the admin port's zero-auth-by-default posture is defensible because it's loopback-only; MAX's web port is not, so this gate has no escape hatch.
- A simple login page (the Vite + React app) posts the password once; `max/server` issues a signed (HMAC, server-held secret) session cookie; every other MAX route (REST + WS upgrade) requires a valid cookie, `401`/WS-close otherwise. Static assets for the login page itself are the only unauthenticated route.
- Document this prominently — the plaintext password file, its rotation story (delete the file + restart `max/server` to force a fresh self-generated value, exactly like `--rotate-admin-key`'s manual equivalent), and that it is not the same secret as the admin key.

### E. The dashboard

- **Layout**: `max` wordmark top-left, top-bar tabs `Keys` / `Profiles` / `Sessions`, per the summary — this **intentionally departs** from `aspec/uxui/interface.md`'s speculative "left navigation" sketch (see Codebase Integration).
- **Keys tab**: CRUD over `GET`/`POST`/`DELETE /admin/v1/keys` — list, create (name + profile picker sourced from `GET /admin/v1/profiles`), revoke. Mirrors `baectl`'s `create key`/`list keys`/`delete key` one-for-one. The plaintext key is shown exactly once at creation with an explicit "copy this now" warning, matching the API's and `baectl`'s own one-time-display convention. MAX's own auto-provisioned `max-observer-<profile_id>` keys (section C) appear in this same list like any other key — badge them distinctly (e.g. "auto-created by MAX") so an operator recognizes them rather than being confused by keys they didn't create (see Edge Cases for why this matters for profile deletion).
- **Profiles tab**: CRUD over `GET`/`POST`/`PUT`/`DELETE /admin/v1/profiles`. The create/edit form's `primary_provider`/`fallback_providers`/`mcp_servers` fields are pickers populated from `GET /admin/v1/providers` and `GET /admin/v1/mcp-servers` (never free-text — a typo'd registry name is only caught at session time today, per `aspec/work-items/0005-parallel-client-handling.md`; a picker prevents the typo class entirely at the UI layer, which is a strict UX improvement over `baectl`'s free-text flags). `allowed_tools` remains free-text (client-declared tool names aren't server-known).
- **Sessions tab**: list from `GET /admin/v1/sessions` (default filtered to `state=open`, with a toggle to show closed/error too), each row linking to the event-graph detail view (section C's bridge) driven by `GET /admin/v1/sessions/{id}/events` (history) + the WebSocket (live tail, only if still `open`).
- **The event graph**: nodes are events (`id`, `event_type`, `client_key_id`, `created_at`), laid out in chronological sequence; color/shape-coded by `event_type` category (client turn, provider request/response, tool call/result, MCP request/response, session lifecycle, join/driver-register — the 14 variants in `server/src/events.rs`'s `EventType::ALL`). Clicking (or, per the accessibility note below, tab-plus-enter-ing) a node opens a detail panel with the full `payload` pretty-printed. New nodes append live as WebSocket messages arrive.
- **Accessibility**: `aspec/uxui/interface.md` already commits this project to "keyboard-navigable ... no interaction that requires a pointer" and WCAG AA contrast for any future dashboard — the event graph's node-click-for-detail interaction must have a keyboard equivalent (focusable nodes, `Enter`/`Space` opens the same detail panel), not just a mouse/touch affordance.

### Container image and dual-process startup

- New `Dockerfile.max` at the repo root, alongside the existing `Dockerfile` (which — once work item 0004 lands — already builds `baesrv` + `baectl`). `Dockerfile.max` independently re-does that same Rust build stage (accepting the small duplication, consistent with this project's "independent components, no lockstep" philosophy — `aspec/architecture/design.md` Principle 3) **plus** a Node build stage compiling `max/web` (Vite `dist/`) and `max/server` (TypeScript → JS), landing all of it — `baesrv`, `baectl`, the compiled `max/server`, and `web/dist` — on the same `debian:bookworm-slim` runtime base as today, with Node 22 installed the same way `Dockerfile.dev` already does (nodesource `setup_22.x`).
- **One new entrypoint script, `docker/bae-max-entrypoint.sh`**, replacing the plain `ENTRYPOINT ["baesrv"]` (`Dockerfile:35`) for this image variant only. It must start both `baesrv` and `node .../max/server` as background children, forward `SIGTERM`/`SIGINT` to both, and — critically — **exit non-zero and kill the other child the instant either process dies**, so a crashed `baesrv` doesn't leave a "healthy-looking" container with only `max` still running (or vice versa):
  ```bash
  #!/bin/bash
  set -e
  baesrv &
  BAESRV_PID=$!
  node /usr/local/lib/max/server/index.js &
  MAX_PID=$!
  trap 'kill -TERM $BAESRV_PID $MAX_PID 2>/dev/null' TERM INT
  wait -n "$BAESRV_PID" "$MAX_PID"
  EXIT_CODE=$?
  kill -TERM $BAESRV_PID $MAX_PID 2>/dev/null
  wait "$BAESRV_PID" "$MAX_PID" 2>/dev/null
  exit $EXIT_CODE
  ```
  `wait -n` is bash-specific (not POSIX `sh`/`dash`, `debian:bookworm-slim`'s default `/bin/sh`) — the shebang must be `#!/bin/bash` and the runtime stage must not assume bash is present without checking (verify it is; install explicitly if not).
- New ports: `BAE_MAX_ADDR` (default `0.0.0.0:3000`) for `max/server`'s HTTP+WS surface, `EXPOSE`d alongside the existing `8080`. The admin port (`8081`) stays loopback-only and **not** newly exposed — `max/server` reaches it over `127.0.0.1:8081` from inside the same container, exactly as `baectl` does; this work item adds no new network exposure to the admin port itself, only to `max`'s own new port (which is why section D's auth gate exists).
- Same data volume (`/var/lib/bae`), no new volume — the observer-keys file and MAX's password file both live there, alongside the admin key file, on the volume operators already mount.

## Edge Case Considerations:

- **MAX's own `max-observer-<profile_id>` key blocks profile deletion.** `DELETE /admin/v1/profiles/{id}` returns `409 profile_in_use` while any active key references it (established behavior, per `aspec/work-items/0004-baectl-cli.md`'s edge cases). Because MAX auto-provisions and never auto-revokes an observer key for every profile it has ever been asked to observe, a profile MAX has touched can become **permanently** undeletable until an operator notices and manually revokes MAX's own key — surprising if the operator doesn't recognize it as MAX's. Mitigated by the Keys-tab badge (section E) and by documenting this plainly; this work item does not attempt automatic cleanup (there's no reliable signal for "MAX no longer needs to observe this profile").
- **Observing a session, then it closes mid-view.** The upstream `session.subscribe` NDJSON stream ends (server-side `Err(RecvError::Closed)` in `drive_subscribe`) with no explicit terminal frame beyond whatever `session.close`/`session.error` event was already forwarded. `max/server` must close the downstream browser WebSocket cleanly and the UI must show "session ended," not hang waiting for more events.
- **Lagged subscriber.** Covered in Implementation Details (section C) — must reconnect with `since_event_id`, not silently stop.
- **Multiple browser viewers on the same live session.** Fan out one upstream `session.subscribe` to N downstream WebSocket clients (section C) — must not mint a new session key or open a second upstream stream per browser tab.
- **`max/server` restarts mid-observation.** In-memory `session_id -> session_key` cache is lost; a reconnecting browser causes a fresh `join` (new session key, new `session.join` event logged) — acceptable and visible in the event log, not silently hidden. The persisted per-profile observer-key file survives the restart, so at least profile-level key provisioning isn't repeated.
- **Viewing a `closed`/`error` session MAX was never connected to while it was open.** Handled entirely by the new admin-port `GET /admin/v1/sessions/{id}/events` (section B) — no `join` is attempted (it would `409 session_closed` anyway), and full history is still readable.
- **No profiles/providers/MCP servers configured yet.** Profiles tab shows an empty state explaining what a profile is and pointing at the create form (per `aspec/uxui/interface.md`'s existing "every empty list explains what the resource is" convention); the create-profile form's provider/MCP pickers are empty and creation is disabled with an explanatory message rather than allowing a profile that can never resolve a provider.
- **`BAE_MAX_PASSWORD_FILE` exists but MAX's login secret was rotated by deleting the file and restarting.** Every existing session cookie becomes invalid (signed with the old secret) — logged-in browsers are forced to re-authenticate; document this as the expected rotation behavior, mirroring `--rotate-admin-key`'s "old plaintext no longer authenticates" posture.
- **A client-side tool call the session's real driver is waiting on, observed by MAX.** MAX never dispatches tools and never sends `tool_result` — it only ever watches; a `Paused` turn looks, from MAX's observer view, like any other `tool.call` event with no forthcoming `tool.result` until the actual owning driver supplies one (or `BAE_TURN_TIMEOUT` abandons it) — MAX must not attempt to "help" by dispatching anything.
- **Large sessions (thousands of events).** The event graph must paginate/virtualize rendering rather than mount every node at once — especially load-bearing on mobile/tablet given the summary's mobile-friendliness requirement and typically tighter memory/CPU budgets.
- **Mobile/tablet viewport for the event graph specifically** (the one view most tempted to assume a wide desktop canvas): must remain usable — pan/zoom or a list-first fallback view — not just shrink proportionally into illegibility.

## Test Considerations:

- **Unit — `max/server` admin-client request/response mapping**: request bodies built for `create profile`/`create key`/`replace profile` match `docs/reference/admin-api.md`'s documented shapes field-for-field, mirroring `baectl`'s own equivalent unit tests (`aspec/work-items/0004-baectl-cli.md`'s "Unit — admin request building").
- **Unit — observer-key provisioning**: first observe request for a profile with no existing entry in `max-observer-keys.json` calls `POST /admin/v1/keys` and persists the result; a second request for the same profile reads the cached entry and makes no admin-API call.
- **Unit — session-key caching**: first `join` for a session mints and caches a session key; a second concurrent observer for the same session reuses the cached key and does not call `join` again.
- **Unit — lagged-subscriber recovery**: a mocked `-32000 "lagged"` notification triggers exactly one `session.subscribe` reconnect with `since_event_id` set to the last forwarded event id.
- **Integration — new admin endpoints**: boot the real admin router (server crate's existing ephemeral-port/temp-DB test harness) and drive `GET /admin/v1/sessions` (pagination, `?state=` filter, empty-result case) and `GET /admin/v1/sessions/{id}/events` (matches `GET /api/v1/sessions/{id}/events`'s shape byte-for-byte for the same session; works against a `closed`/`error` session with no session key involved at any point; `404` for a nonexistent id).
- **Integration — full observe flow**: boot the real server + `max/server` against it; open a session and send messages as an ordinary driver (via a raw HTTP client, not `max`); have `max/server` observe it; assert the browser-facing WebSocket receives every event in order and that `max/server` never issues `session.registerDriver` or `session.sendMessage` (assert via request-log inspection of the mock/real server, not just absence of errors).
- **Integration — fan-out**: two simulated browser WS clients observing the same session through one `max/server` process; assert only one upstream `join`/`subscribe` occurred (one `session.join` event logged) and both clients received every event.
- **Regression — profile-deletion interaction**: after MAX observes at least one session under a profile, assert `DELETE /admin/v1/profiles/{id}` still returns `409 profile_in_use` (unchanged existing behavior) and that MAX's own key is visibly distinguishable in `GET /admin/v1/keys`'s response (via its `max-observer-` name prefix) so the regression test also documents the mitigation, not just the constraint.
- **Regression — MAX auth gate**: any MAX route (REST or WS upgrade) other than the login page/static assets rejects a request with no/invalid session cookie; correct password grants a cookie that then authorizes; wrong password is rejected without revealing whether the password was close.
- **Regression — dual-process entrypoint**: build (or approximate in CI without a full image build) the `bae-max` entrypoint script's signal-forwarding behavior — kill one child process, assert the script forwards `TERM` to the other and exits non-zero; send `SIGTERM` to the script itself, assert both children receive it and the script exits cleanly.
- **Regression — image smoke test**: `make image-max` builds successfully; running it exposes both `GET /healthz` (port 8080, unauthenticated liveness) and `max/server`'s own health route (port 3000) as healthy; killing the `baesrv` process inside a running container causes the whole container to exit (proving the entrypoint's failure-propagation, not just that both processes started once).
- **E2E (browser-driven, e.g. Playwright)**: Keys/Profiles CRUD round-trips through the actual UI; opening a live session's graph view and seeing nodes append as a scripted driver sends messages; clicking (and, separately, keyboard-activating) a node opens its detail panel with the correct payload; responsive layout smoke-checked at common mobile and tablet breakpoints.
- All new server-side tests remain fully offline (`make test-server`, `make test-max`), matching every existing component's test posture — no real network calls, no real provider keys.
