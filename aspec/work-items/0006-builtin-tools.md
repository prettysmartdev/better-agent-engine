# Work Item: Feature

Title: builtin tools
Issue: issuelink

## Summary:
- This work item adds several builtin tools to the server and client harnesses.

First, sandboxes:

Both the baesrv server and each of the client harnesses must learn how to launch sandboxed environments in which to run shell commands. There will be two sandbox drivers available (which must be modular implementations of a common 'SandboxDriver' trait/interface. Docker and Apple Containers.

For the server side, the 'profile' object must gain a new field called 'availableSandboxes' with a list of container image names that are available to be launched. When a profile is created with a list of available images, the server should spawn a background task that ensures all of the named images are available, pull any that do not exist, and log when they are available and/or any errors. After a client associated with that profile connects as a session driver, the server should send an informative notification letting the client know which sandbox images are available for use.

The client should then be able to send a command (like client.startRemoteSandbox) with one of the available images as an option, which should indicate to the server that a sandbox should be started using the configured sandbox provider.

Similarly, the client harness libraries should have builtin sandbox tools that harness developers can optionall bind to their custom harness which advertise either full-shell or specific shell commands as tools to the LLM. There should be both local-execution and remote-execution variants (i.e. run_shell_command(remote|local) which tells the LLM it can execute any shell command it wants AND run_shell_named(client_defined_shell_command, local|remote) which tells the LLM there is a specific shell command it can run (with options for interpolating specific arguments). The HARNESS then decides (at the time the LLM sends the tool call) wether a specific call should be executed in a local or remote sandbox.

If the harness routes a run_shell_command or run_shell_named to its LOCAL sandbox, it runs docker exec/container exec against its local container engine. If the harness routes one of these calls to the REMOTE sandbox, it sends a message with the fully-formed shell command (interpolations already complete) to the server and CHOOSES wether the server immediately routes the command output directly back to the LLM as the tool result OR returns the tool result to the harness for further processing before sending back the tool result. The client SDKs must make all of this simple for the user with "nice" developer-facing APIs/objects/options. 

The client harnesses should also each gain 3 builtin tools that developers can choose to mount to their harnesses: read file, write file, explore files (which returns file lists). The harness developer must CHOOSE the security constraints for each of these tools when creating/attaching them, such as: allowed directories, allowed/disallowed filenames/extensions, and allow/disallow regexes for filenames.

Ensure all implementations across server and client harnesses are MODULAR, SECURE, and CONSISTENT with one another.

## User Stories

### User Story 1:
As an: Agent Developer

I want to:
bind a `run_shell_command`/`run_shell_named` tool to my harness and, at the moment the LLM actually calls it, decide per-call whether it runs against my local Docker/Apple Container engine or against the sandbox the server started for this session — without hand-rolling any subprocess or wire-protocol code myself

So I can:
give an agent real shell/execution ability with a security boundary I control (which images, which commands, local vs. server-hosted), instead of either giving the model unrestricted host access or writing a bespoke sandboxing layer for every harness

### User Story 2:
As a: Platform Operator

I want to:
declare a profile's `availableSandboxes` image list once and have the server pre-pull/verify every image in the background as soon as the profile exists (not on the critical path of a client's first session), logging success or failure per image, and have connecting clients get told which images are actually ready to use

So I can:
avoid a client's first `startRemoteSandbox` call stalling on a multi-hundred-megabyte image pull, and get clear operational visibility (via logs) into which sandbox images are misconfigured or unavailable on this host before an agent developer ever hits the failure

### User Story 3:
As an: Agent Developer

I want to:
mount the builtin `read_file`/`write_file`/`explore_files` tools to my harness while declaring my own directory allowlist, extension allow/deny lists, and filename regexes — so an LLM-driven file tool can never touch a path I didn't explicitly permit

So I can:
give an agent filesystem access scoped to exactly the project directory (or subset of it) I intend, reusing a vetted, consistent implementation across Rust/TypeScript/Python instead of re-writing path-traversal-safe validation myself in each language


## Implementation Details:

This work item is almost entirely new surface (unlike WI 0005, nothing here exists yet). It splits into six mostly-independent pieces: (A) a server-side `SandboxDriver` abstraction over Docker and Apple Containers, (B) the profile `availableSandboxes` field plus background image provisioning and a driver-connect notification, (C) the `session.startRemoteSandbox` control-plane call and the server-side remote-exec dispatch path, (D) client-harness builtin sandbox tools (local and remote, full-shell and named-command), (E) client-harness builtin file tools with developer-chosen security constraints, and (F) cross-cutting wiring (event types, config, docs). All server pieces follow the module-separation precedent `engine/mcp.rs` already established for "external, potentially slow, subprocess/network-backed capability the engine dispatches into" (see that file's `# Why hand-rolled` and `# Lifecycle` doc comments) — sandboxes get the same treatment rather than pulling in a Docker Engine API client SDK.

### A. `SandboxDriver` trait + Docker / Apple Containers implementations (server)

New module `server/src/engine/sandbox.rs`, mirroring `engine/mcp.rs`'s shape: a small hand-rolled surface (spawn a subprocess per call, no SDK), a closed set of transports, and every failure mode captured explicitly.

```rust
/// A running sandboxed container, opaque to callers beyond its id.
pub struct SandboxHandle { pub id: String, pub image: String }

pub struct ExecResult { pub stdout: String, pub stderr: String, pub exit_code: i32 }

#[derive(Debug)]
pub enum SandboxError {
    /// This host/platform cannot run this driver (e.g. Apple Containers on Linux).
    Unsupported(String),
    /// Pulling/inspecting an image failed.
    Image { image: String, detail: String },
    /// Starting, execing into, or stopping a container failed.
    Runtime { detail: String },
}

#[allow(async_fn_in_trait)]
pub trait SandboxDriver: Send + Sync {
    /// Idempotent: inspect `image` locally; pull it if absent. Called both by
    /// the profile-create background task and defensively before `start`.
    async fn ensure_image(&self, image: &str) -> Result<(), SandboxError>;
    /// Start a long-lived container from `image` (keep-alive command, no
    /// published ports — exec is the only interaction surface) and return its
    /// handle.
    async fn start(&self, image: &str) -> Result<SandboxHandle, SandboxError>;
    /// Run one shell command inside an already-started container and capture
    /// stdout/stderr/exit code. Never interprets the command string itself —
    /// the caller is responsible for any interpolation before this point.
    async fn exec(&self, handle: &SandboxHandle, command: &str) -> Result<ExecResult, SandboxError>;
    /// Stop and remove the container. Idempotent; safe to call on an already-gone id.
    async fn stop(&self, handle: &SandboxHandle) -> Result<(), SandboxError>;
}
```

- `DockerDriver`: shells out to the `docker` CLI via `tokio::process::Command` (same primitive `engine/mcp.rs::StdioConn::spawn` already uses), exactly the way the rest of the engine avoids heavy SDKs: `docker image inspect <image>` (exit 0 ⇒ present) → `docker pull <image>` on miss; `docker run -d --rm <image> sleep infinity` capturing the printed container id for `start`; `docker exec <id> sh -c <command>` for `exec`; `docker stop <id>` for `stop`. Every non-zero exit maps to `SandboxError::Image`/`SandboxError::Runtime` carrying stderr (truncated), never panics on a missing `docker` binary — that surfaces as `Runtime` and is logged/skipped exactly like an MCP connect failure.
- `AppleContainerDriver`: identical shape against Apple's `container` CLI (`container images inspect`, `container run -d`, `container exec`, `container stop`). Its constructor checks `cfg!(target_os = "macos")` (or, in a container/CI environment, `std::env::consts::OS`) and returns `SandboxError::Unsupported` immediately if instantiated on a non-macOS host — a clear, structured failure rather than a confusing subprocess-spawn error, so an operator misconfiguring `BAE_SANDBOX_DRIVER=apple-container` on a Linux host gets one unambiguous log line at startup.
- Driver selection is one new `BAE_SANDBOX_DRIVER` env var (`docker` default, or `apple-container`), read in `server/src/config.rs` alongside `BAE_TURN_TIMEOUT` (`server/src/config.rs`, see `DEFAULT_TURN_TIMEOUT_SECS`), and turned into a boxed `Arc<dyn SandboxDriver>` once at startup in `cli.rs::run_serve` (a new `build_sandbox_driver(&config) -> Arc<dyn SandboxDriver>` next to `open_store`). One driver, chosen server-wide — not per-profile — since it reflects what container engine is actually installed on *this host*, whereas `availableSandboxes` (below) is the per-profile *image allowlist* on top of that host-wide driver.
- `AppState` (`server/src/api/mod.rs`) gains `pub sandbox_driver: Arc<dyn SandboxDriver>`, following the exact `Arc<...>`, constructed-once-at-startup, cheap-to-clone shape `mcp_registry`/`provider_registry` already use.

### B. Profile `availableSandboxes`, background provisioning, driver-connect notification (server)

- New migration `server/src/store/migrations/0007_profiles_sandboxes.sql`, registered in `store/migrations.rs::MIGRATIONS` as version 7: `ALTER TABLE profiles ADD COLUMN available_sandboxes TEXT` — same "JSON blob in a TEXT column" convention as `mcp_servers`/`allowed_tools` (`store/migrations/0003_profiles.sql`), not a new table, and forward-only per `store/migrations.rs`'s doc comment.
- `store/profiles.rs::ProfileRecord`/`ProfileInput` gain `available_sandboxes: Value` (a JSON array of image-name strings); `SELECT_COLS`, `create`, `replace`, and `row_to_record` are extended the same mechanical way the existing five JSON-blob columns already are — no new query shape.
- `api/admin/profiles.rs::CreateProfile` gains `available_sandboxes: Option<Value>`, validated with the existing `require_string_array` helper (reused verbatim, exactly as the Codebase Integration note on WI 0005 already calls for reusing it rather than writing a near-duplicate). `profile_view` adds `"available_sandboxes": p.available_sandboxes`.
- **Background provisioning.** In `admin/profiles.rs::create` (and `replace`, since an admin may add new images to an existing profile), after the DB write succeeds, `tokio::spawn` a detached task — never awaited by the HTTP handler, so profile creation returns immediately regardless of image size or registry latency — that calls `state.sandbox_driver.ensure_image(image)` for every declared name, sequentially (parallel pulls of large images competing for one host's disk/network bandwidth is not obviously desirable output; keep it simple and serial for v1). Each image logs exactly one of:
  - `tracing::info!(profile_id, image, "sandbox image already available")`
  - `tracing::info!(profile_id, image, "sandbox image pulled successfully")`
  - `tracing::error!(profile_id, image, error = %e, "failed to ensure sandbox image")`
  This mirrors `resolve_mcp_servers`'s per-name log-and-continue posture (`api/client/sessions.rs:418-466`) but fires once at profile-write time instead of every session open, since image availability is a slow, profile-scoped concern, not a per-session one.
- **In-memory status tracking**, so the driver-connect notification (below) can report real status rather than just echoing the configured list: `AppState` gains `pub sandbox_status: Arc<Mutex<HashMap<String, HashMap<String, SandboxImageStatus>>>>` keyed `profile_id -> image -> status`, where `SandboxImageStatus` is `Pending | Available | Error(String)`. The background task in `admin/profiles.rs` updates this map as each `ensure_image` call resolves; it starts every declared image at `Pending` synchronously (inside the same transaction-adjacent code, before spawning) so a client connecting the instant after profile creation sees `pending` rather than nothing. Like `mcp_sessions`/`drivers`/`turn_gates`, this is in-memory only, rebuilt from a fresh `Pending` state for every image on every profile at server restart (a restart re-triggers `ensure_image` for every profile with a non-empty `available_sandboxes` at startup, so status is never permanently stale — see Codebase Integration).
- **Driver-connect notification, scoped strictly to this session's own profile.** When a client registers as a driver (`session.registerDriver`, `api/client/rpc.rs:152` region) and the session's profile has a non-empty `available_sandboxes`, log (and thus broadcast, via the existing `broadcast::insert_and_publish` choke point) a new `EventType::SandboxAvailable` (`session.sandbox.available`) event: `{"images": [{"name": "python:3.12", "status": "available"}, {"name": "node:22", "status": "pending"}, ...]}`. **Critically, the `images` list must be built by iterating *this session's profile's own* `profile.available_sandboxes` array and looking up each name's status in `state.sandbox_status.get(&profile.id)` — never by flattening or scanning `sandbox_status` across all profiles.** `sandbox_status` is a server-wide map covering every profile that has ever declared images; a different profile on the same running server may declare completely different (or overlapping-but-differently-permissioned) images, and those must never appear in this notification even if their pull status happens to be known. The correct implementation reads as "for each name in `profile.available_sandboxes`, look up `sandbox_status[profile.id][name]`" — a per-profile projection, not a global one — and this scoping is exactly what part C's `startRemoteSandbox` enforcement below must also honor, so the notification and the enforcement never disagree about which images are usable in this session. This reuses the exact mechanism `EventType::SessionDriverRegistered` already uses — no new transport concept, just a new event type logged at the same call site, immediately after the driver-registration event. Every other participant (per the WI 0005 cross-visibility model) also observes it, which is harmless and consistent with "all driver clients receive all events" — because the payload was already scoped to the *session's* (single) profile before it was ever broadcast, cross-profile leakage cannot happen via multi-driver fan-out either.

### C. `session.startRemoteSandbox` / `session.stopRemoteSandbox` + remote-exec dispatch (server)

**Lifecycle events.** A remote sandbox's life is modeled as a proper state machine, mirroring how `session.open`/`session.close`/`session.error` already model a session's: a *request* event fires the moment the server accepts the intent, and a distinct *resolution* event fires once the underlying `SandboxDriver` call actually completes — success or failure. Six new event types carry this (all under the `session.sandbox.*` prefix, consistent with `session.open`/`session.join`/`session.driver.register`):

| Event | Fired when |
|---|---|
| `session.sandbox.start` | `startRemoteSandbox` request accepted (image validated against the allowlist) and about to call the driver |
| `session.sandbox.running` | `SandboxDriver::start` succeeded; the sandbox is up and its handle retained |
| `session.sandbox.stop` | `stopRemoteSandbox` request accepted (or session close triggering an implicit stop) and about to call the driver |
| `session.sandbox.stopped` | `SandboxDriver::stop` succeeded |
| `session.sandbox.error` | any driver call (`ensure_image`, `start`, `exec`, `stop`) failed, at any point in the lifecycle |

(The `sandbox.request`/`sandbox.response` dispatch events introduced below are a separate concern — one `Auto`-mode tool-call round trip, not sandbox lifecycle — and deliberately keep the *un*-prefixed `sandbox.*` shape that mirrors `mcp.request`/`mcp.response`, which are likewise unprefixed. The `session.sandbox.*` prefix is reserved for lifecycle state transitions.)

- New JSON-RPC method `session.startRemoteSandbox` (params `{"image": string}`), dispatched in `rpc()`'s method match (`api/client/rpc.rs:133-151` region) alongside `registerDriver`/`sendMessage`. Requires the caller to already be a registered driver (same `-32001` gate as `sendMessage`). **Validates `image` against *this session's own profile's* `available_sandboxes` only** — the same profile the session was opened/joined against, loaded the same way `sessions.rs::create`/`join` already load it (`profiles::get(c, &session.profile_id)`) — never against any other profile's list, even one belonging to the same admin or the same running server. An image absent from *this profile's* list is rejected with a JSON-RPC error (`-32011 sandbox_image_not_allowed`) regardless of whether that image is legitimately declared on some other profile: **the profile bound to the session is the only trust boundary for what that session may launch**, exactly analogous to `allowed_tools` gating client tool declarations, and exactly the same scoping part B's driver-connect notification must honor. Logs `session.sandbox.start` (payload `{"image"}`) before calling the driver.
- Calls `state.sandbox_driver.start(image)` (calling `ensure_image` first defensively, in case the background provisioning task hasn't finished or the server restarted since — the "startup re-triggers ensure_image" path referenced in part B).
  - On success: retains the handle on `AppState` keyed by session id — `pub sandboxes: Arc<Mutex<HashMap<String, Arc<tokio::sync::Mutex<SandboxHandle>>>>>`, the same `Arc<Mutex<HashMap<SessionId, Arc<tokio::sync::Mutex<_>>>>>` shape `mcp_sessions` already uses (an `exec` call holds the lock across an `.await`, same reason). Logs `session.sandbox.running` (payload `{"image", "sandbox_id"}`) and returns the terminal RPC response `{"sandbox_id": "...", "image": "...", "started_at": "..."}`.
  - On failure: logs `session.sandbox.error` (payload `{"image", "phase": "start", "detail"}`), retains no handle, and returns a JSON-RPC error (`-32012 sandbox_start_failed`).
  - One remote sandbox per session (not per driver): like MCP servers, the remote sandbox is session-wide shared infrastructure any driver's turn may use, not a private per-driver resource — call this out explicitly since it is a deliberate scoping choice, not an oversight (see Edge Cases).
- New JSON-RPC method `session.stopRemoteSandbox` (params `{}`, operating on the session's one retained sandbox), same driver-registration gate. Rejects with a clear error if no sandbox is currently running for this session (`-32013 sandbox_not_running`). Logs `session.sandbox.stop` (payload `{"image", "sandbox_id", "reason": "explicit"}`) before calling the driver, then `session.sandbox.stopped` (same payload shape) on success or `session.sandbox.error` (`phase: "stop"`) on failure; the entry is removed from `AppState.sandboxes` regardless of whether the driver call reported success, so a failed stop never leaves a phantom handle other calls could still dispatch against.
- Session close (`api/client/sessions.rs::close`) triggers the identical stop sequence — the same helper function `session.stopRemoteSandbox`'s handler calls internally, so the two code paths cannot drift — with `"reason": "session_close"` instead of `"explicit"` in the `session.sandbox.stop` payload, in the same teardown function that already drops the broadcaster entry, MCP session, and multi-client runtime state (`sessions.rs:701-705` region). A session with no started sandbox performs no-op teardown here, same as MCP teardown on a session with no connected servers.
- **Remote-exec dispatch, and the auto/manual choice.** The summary's "the server CHOOSES whether the server immediately routes the command output directly back to the LLM as the tool result OR returns the tool result to the harness for further processing" is a per-*tool-binding* choice the harness developer makes when they construct the tool (see part D), not a per-call flag — it determines which of two server code paths a remote shell tool's `tool_use` takes:
  - **Manual (default)**: the shell tool is declared as an ordinary client-side tool (the existing `tools` array in `POST /api/v1/sessions`/`join`, unchanged shape). `run_turn` treats it exactly as today: `Outcome::Paused`, the assistant turn (with its `tool_use`) is handed back to the client. The client harness's tool handler, on the local-vs-remote decision it already made, issues a **new, separate, non-turn RPC method** `session.execRemoteSandbox` (params `{"command": string}`, no session-loop involvement at all — a synchronous utility call much like `registerDriver`) to run the fully-interpolated command in the session's live remote sandbox and get back `{stdout, stderr, exit_code}`. A driver-call failure here also logs `session.sandbox.error` (`phase: "exec"`) so a broken remote sandbox is visible in the event log even when dispatched manually. The harness's own code (a lifecycle hook or the tool handler itself) does whatever post-processing it wants with that raw result, then constructs the `tool_result` content itself and sends it back via the ordinary `session.sendMessage` continuation — identical to how every other client tool works today. This path requires **no run_turn changes at all**.
  - **Auto**: the shell tool is instead declared in a new, parallel per-client list at session open/join — `sandbox_tools: [{"name", "description", "input_schema"}]` alongside `tools` — stored the same per-client-keyed-object way `client_tools` already is (`store/sessions.rs::set_client_tools`), but in a sibling column/key so the two are never confused. `engine/session.rs::run_turn` (`engine/session.rs:118-160` region) gains a third dispatch bucket alongside `client_tool_names` and "everything else is MCP": a `sandbox_tool_names` set built from the acting client's `sandbox_tools` entry. A `tool_use` whose name is in that set is dispatched **server-side**, exactly like an MCP tool: `EventType::SandboxRequest`/`SandboxResponse` events (mirroring `McpRequest`/`McpResponse`) wrap a `state.sandbox_driver.exec(handle, command)` call against the session's retained sandbox handle, the result becomes the `tool_result` appended to the in-memory `history`, and the loop continues (`for _ in 0..MAX_ITERATIONS`) without ever pausing or involving the client — the LLM gets its answer in the same provider round-trip sequence a fully server-dispatched MCP tool would. A failed exec here also logs `session.sandbox.error` (`phase: "exec"`) alongside the error-shaped `sandbox.response`/`tool.result`, so lifecycle visibility is identical regardless of dispatch mode. A sandbox-tool call arriving when the session has no started remote sandbox (client never called `startRemoteSandbox`) is handled exactly like the existing "no MCP server configured for this tool" case (`engine/session.rs:409-419`): an error-shaped `tool.result`, turn continues.
  - Both paths dispatch into the *same* `SandboxDriver::exec`; the only difference is which layer (client harness vs. `run_turn`) owns constructing the `tool_result` and whether the turn pauses. This mirrors, deliberately, the existing client-tool-vs-MCP-tool dichotomy `run_turn` already implements — sandbox "auto" tools are dispatched exactly like MCP tools, sandbox "manual" tools are dispatched exactly like ordinary client tools, so no new dispatch primitive is invented, only a new *source* (the sandbox driver instead of an MCP connection) feeding an existing shape.

**`session.reportLocalSandbox` — client-originated local-sandbox lifecycle events.** The remote lifecycle above is entirely server-authored (the server is the only party that knows whether its own sandbox is running). A **local** sandbox — started by a client harness against its own Docker/Apple Container engine — is invisible to the server unless the client tells it. New JSON-RPC method `session.reportLocalSandbox` (params `{"state": "running" | "stopped" | "error", "image": string, "container_id": string | null, "detail": string | null}`), same driver-registration gate as the other session-scoped methods (any registered driver may report its own local sandbox activity — it does not have to be the turn's current owner, since local sandbox lifecycle is orthogonal to turn ownership). The server does **not** validate `image` against `available_sandboxes` here — a local sandbox is the harness developer's own local trust decision (part D), not a server-governed resource — it only maps `state` to the matching `EventType` (`SandboxRunning`/`SandboxStopped`/`SandboxError`) and logs it with payload `{"dispatch": "local", "image", "container_id", "detail"}`, attributed to the caller's `client_key_id`, through the same broadcast choke point every other event uses. This is the mechanism by which "client harnesses ALSO emit `session.sandbox.running`/`session.sandbox.stopped`/`session.sandbox.error` events" becomes visible to every subscriber, not just the reporting client. The method is deliberately named/scoped to *local* only (there is no `"scope"` parameter to set) so a client can never forge a *remote* sandbox lifecycle event under this path — the remote lifecycle stays exclusively server-authored via the `start`/`stop`/exec code paths above.

### D. Client-harness builtin sandbox tools

A **prerequisite, cross-SDK-inconsistent gap** this work item must fix first: `client-rust/src/tool.rs`'s `ToolHandler` is `Box<dyn Fn(Value) -> Result<Value, BoxError> + Send + Sync>` — **synchronous**, invoked with a plain (non-`.await`) call in `harness.rs::run_loop` (`tool.call(call.input.clone())`, no `.await`, `harness.rs:124-129`). A remote-sandbox tool handler fundamentally needs to make an async HTTP round-trip (the new `session.execRemoteSandbox` RPC); a synchronous closure cannot do that without blocking the async runtime thread. `client-typescript/src/tool.ts`'s `ToolHandler` (`Content | Promise<Content>`) and `client-python/src/bae_py/tool.py`'s `ToolHandler` (`Union[Content, Awaitable[Content]]`) **already** support async handlers — only Rust needs a breaking change. Change `client-rust`'s `ToolHandler` to `Box<dyn Fn(Value) -> Pin<Box<dyn Future<Output = Result<Value, BoxError>> + Send>> + Send + Sync>` (or an `async fn`-returning-closure via a small helper trait, implementer's choice — the point is `Tool::call` becomes `async fn call(&self, input: Value) -> Result<Value, BoxError>` and `harness.rs::run_loop`'s call site adds `.await`). This is a deliberate, acceptable breaking change to `client-rust`'s public `Tool::new` signature (alpha status, same posture WI 0003/0005 already established for wire-protocol changes) and every existing synchronous tool handler in `client-rust`'s tests/examples needs a trivial `async move { ... }` wrap.

With that prerequisite in place, add one new module per SDK (`client-rust/src/sandbox.rs`, `client-typescript/src/sandbox.ts`, `client-python/src/bae_py/sandbox.py`), each exposing:

- A local-container-engine abstraction that is a **full mirror of the server's `SandboxDriver`** (`ensure_image`/`start`/`exec`/`stop` — not just `exec`), run by shelling out to the local `docker`/`container` CLI exactly like the server's driver does (same rationale: no SDK dependency, subprocess-based, consistent behavior). It needs the full lifecycle, not just `exec`, because the client must now be able to report `running`/`stopped`/`error` transitions to the server (below) — an ad hoc "one-shot `docker exec`" with no tracked container identity would have nothing to report a `container_id` for. Reusing the exact same two-variant trait/interface shape as the server (`SandboxDriver` with `Docker`/`AppleContainer` implementations) is what "modular... and CONSISTENT with one another" in the summary calls for — the *interface* is duplicated per-language (Rust/TS/Python each need their own idiomatic version) but the *behavior and CLI invocations* must match the server's driver byte-for-byte in intent (same commands, same flags), so a harness developer's mental model transfers directly between "the sandbox my client started locally" and "the sandbox the server started remotely."
- **Local sandboxes automatically report their own lifecycle to the server**, so it is encoded in the session's event log and visible to every subscriber — not just something a harness developer can forget to wire up. Concretely: the local driver abstraction (previous bullet) is managed internally by the `run_shell_command`/`run_shell_named` constructors when built with `SandboxTarget::Local`, and around every `start`/`exec`/`stop` call into the local driver, the constructed `Tool`'s handler also issues `session.reportLocalSandbox` (part C) on the same `Session`:
  - the local container starts successfully (lazily, on the first local-target tool call for a given image, unless the developer starts it eagerly via an explicit `Session::start_local_sandbox(image)` call) → report `state: "running"` with the local `container_id`;
  - the container is stopped (explicitly via `Session::stop_local_sandbox()`, or automatically when `Session::close()` runs — each SDK's `close()` must now also stop any still-running local sandboxes it started, mirroring how the server stops its own remote sandbox at session close) → report `state: "stopped"`;
  - any local driver call fails (image pull, start, exec, or stop) → report `state: "error"` with `detail` set to the failure message, in addition to whatever in-band tool-result error the failed `exec` itself produces (the report is pure telemetry into the event log; it never substitutes for the tool call's own error handling).
  This reporting is automatic/built-in specifically so a harness developer gets it "for free" the moment they bind a local sandbox tool, matching the "nice developer-facing APIs" goal already stated for this work item — there is no separate opt-in step.
- A small builder for where a shell tool's commands run and how remote results get handled:
  ```rust
  pub enum SandboxTarget {
      Local { image: String },
      Remote,
  }
  pub enum RemoteMode {
      /// Server dispatches and continues the turn itself (see part C, "auto").
      Auto,
      /// Client fetches raw output via `execRemoteSandbox` and builds the
      /// tool_result itself (see part C, "manual"); the given closure receives
      /// `ExecResult` and returns the tool_result content.
      Manual(Box<dyn Fn(ExecResult) -> Value + Send + Sync>),
  }
  ```
  and two constructors:
  - `run_shell_command(target: SandboxTarget, remote_mode: RemoteMode) -> Tool` — declares a single tool (fixed name `run_shell_command`, developer can override the description) whose `input_schema` requires one `command: string` field; the LLM can request *any* shell command. For `SandboxTarget::Local`, the handler shells directly into the local container engine; for `Remote` + `RemoteMode::Manual`, the handler calls the new `Session::exec_remote_sandbox(command)` method and runs the closure; for `Remote` + `RemoteMode::Auto`, this constructor does **not** produce a client-dispatched `Tool` at all — it instead returns a `SandboxToolDef` destined for the new `sandbox_tools` declaration list (part C), since auto-mode tools are never invoked by the client harness's own loop. This is a meaningful type-level distinction worth making explicit in each SDK's public API (e.g. two returned types, or an enum wrapping either) so a harness developer cannot accidentally register an `Auto` tool through `Harness::with_tool` (which only accepts client-dispatched `Tool`s) and have it silently never fire.
  - `run_shell_named(name, description, command_template: &str, target, remote_mode) -> Tool` (or `SandboxToolDef`, same auto/manual split as above) — declares a tool whose `input_schema` is derived from `{param}`-style placeholders parsed out of `command_template` (each becomes a required string input property), and whose handler substitutes the model-supplied argument values into the template before executing. **Interpolation must be shell-escaped, never naively string-concatenated** — this is a command-injection boundary (see Security below) — use each language's standard shell-quoting primitive (e.g. `shell-escape`-style single-quote wrapping with embedded-quote escaping) on every substituted value before building the final command string handed to `docker exec`/`session.execRemoteSandbox`.
- `Harness::connect`/`Harness::join` (or their TS/Python equivalents) gain `Session::exec_remote_sandbox(command: &str) -> Result<ExecResult, Error>` and `Session::report_local_sandbox(state, image, container_id, detail)` methods — thin wrappers issuing the two new RPC methods — available to any tool handler (or arbitrary application code) via whatever `Session`-reference each SDK's closures already capture. **This is why every sandbox tool built by part D — local *or* remote, manual *or* auto — needs a live `Session`/transport reference, not just the remote ones as originally scoped**: local tools now also call back into the session to report their lifecycle. In Rust this means restructuring `ToolHandler` to receive a lightweight session/transport handle alongside `input` (today's `ToolHandler` only receives `Value`), and it means `run_shell_command`/`run_shell_named` must be constructed *after* `Harness::connect()`/`join()` returns, taking the resulting `Session` (or a cheap clone/handle of its transport) as a constructor argument — an ordering difference from every other builtin tool in this work item (the file tools in part E need no `Session` at all and can be built before `connect()`), worth documenting prominently in each SDK's doc comments so a developer isn't surprised their sandbox tools can't be registered via the pre-`connect()` builder pattern the rest of the harness uses. TS/Python constructors can simply close over the `Session` object directly at tool-build time, since idiomatic usage there already tends to build tools after `connect()` returns; Rust is the SDK where this ordering constraint is most likely to surprise an existing user, so call it out there most prominently.
- The session-open/join request body gains the new optional `sandbox_tools` array (mirrors `tools`) whenever any `Auto`-mode sandbox tool is registered; `Harness::open` (`harness.rs:528-575` region) is extended to also collect and send these.

### E. Client-harness builtin file tools

Pure client-side (no server changes): one new module per SDK (`client-rust/src/files.rs`, `client-typescript/src/files.ts`, `client-python/src/bae_py/files.py`) exposing a security-constraint builder and three `Tool` constructors.

```rust
pub struct FileToolConfig {
    /// Only paths that canonicalize to somewhere under one of these directories
    /// are permitted. Required — an empty list permits nothing (an explicit,
    /// developer-visible "nothing is allowed" rather than an implicit
    /// filesystem-wide default).
    pub allowed_dirs: Vec<PathBuf>,
    /// If non-empty, only these extensions (without the dot) are permitted.
    pub allowed_extensions: Option<Vec<String>>,
    /// These extensions are always rejected, even if `allowed_extensions` would
    /// otherwise permit them (e.g. block `.env` while allowing everything under
    /// `allowed_extensions`).
    pub denied_extensions: Vec<String>,
    /// If set, the path's filename must match this regex.
    pub allow_regex: Option<Regex>,
    /// If set, a filename matching this regex is always rejected.
    pub deny_regex: Option<Regex>,
}

pub fn read_file_tool(config: FileToolConfig) -> Tool;
pub fn write_file_tool(config: FileToolConfig) -> Tool;
pub fn explore_files_tool(config: FileToolConfig) -> Tool; // lists files under an allowed_dir, non-recursive by default with a `recursive: bool` input flag
```

- **Validation order and path-traversal safety** (implemented once as a private `validate_path(config, requested_path) -> Result<PathBuf, ValidationError>` shared by all three constructors, so the three tools cannot drift out of sync): canonicalize the requested path (resolving `..`/symlinks) *before* any allow/deny check — checking the allowlist against the raw, uncanonicalized string is the classic path-traversal bug (`../../etc/passwd` or a symlink escaping the allowed root) this must not repeat. Then: (1) the canonical path must have one of `allowed_dirs`' canonical forms as a prefix; (2) if `denied_extensions` matches, reject; (3) if `allowed_extensions` is set and does not match, reject; (4) if `deny_regex` matches the filename, reject; (5) if `allow_regex` is set and does not match, reject. `write_file_tool` additionally refuses to create a file whose parent directory does not already exist under an allowed dir (no auto-`mkdir -p` traversal outside the sandboxed tree) unless the config explicitly opts in (a `create_parents: bool` config field, default `false`).
- **Failure is an in-band tool result, not an aborted loop.** A validation failure returns `Ok(json!({"error": "path not permitted: <reason>"}))`-shaped content (or each SDK's idiomatic equivalent), not an `Err`/exception — a security-boundary rejection is an expected, foreseeable model behavior (the LLM guessing at a path outside its sandbox), not a program bug, so the model should see it as tool output it can react to and retry from, mirroring how `run_turn` already turns MCP/sandbox failures into error-shaped `tool.result`s rather than aborting the turn (part C). This is the one place a builtin tool constructor must deliberately catch-and-wrap rather than propagate — call this out in each SDK's doc comment so a harness developer copying the pattern for their own tools makes the same choice deliberately, not by accident.
- `explore_files_tool`'s output is a JSON array of `{"path", "is_dir", "size_bytes"}` (relative to whichever `allowed_dirs` entry matched), filtered through the same `validate_path` per discovered entry (so a directory listing can never itself leak an entry that would fail the read/write checks) — an `explore` result must never surface a path a subsequent `read_file` call would then reject as unauthorized, which would be a confusing inconsistency between the three tools.

## Edge Case Considerations:

- **`client.startRemoteSandbox` with an image not in the profile's `available_sandboxes`**: rejected with `-32011 sandbox_image_not_allowed` before any container is started — the admin-declared allowlist is the sole trust boundary; a client can never cause the server to pull/run an arbitrary image of its own choosing.
- **An image is declared on a *different* profile but not on this session's own profile**: also rejected with `-32011 sandbox_image_not_allowed`, identically to an image unknown anywhere on the server — `startRemoteSandbox`'s validation (part C) checks only `session.profile_id`'s own `available_sandboxes`, never any other profile's. Symmetrically, the driver-connect notification (part B) must never list that other profile's image either, even though `AppState.sandbox_status` internally tracks its pull state (because some *other* session's profile provisioned it). This is the specific cross-profile leakage this work item must not introduce, and it must be covered by a dedicated test with two profiles declaring disjoint image sets (see Test Considerations) — a test with only one profile in play cannot catch a scoping bug where the implementation accidentally unions/flattens `sandbox_status` across profiles.
- **A profile's `available_sandboxes` image fails to pull** (typo'd name, private registry auth not configured on the host, disk full): logged as `tracing::error!` by the background task (part B), `sandbox_status` set to `Error(detail)`; the driver-connect notification (part B) reports that image's status as `error` rather than omitting it, so the harness developer sees the failure surfaced in-band rather than silently getting a `startRemoteSandbox` rejection later with no context. `startRemoteSandbox` for that image still attempts `ensure_image` again defensively (part C) and fails with a clear `sandbox_start_failed` error if the underlying problem persists (e.g. still no registry auth) — never silently succeeds against a stale/partial pull.
- **Server restarts with sandboxes already running from a prior process**: `AppState.sandboxes`/`sandbox_status` are in-memory only (same posture as `mcp_sessions`); a restarted server has no record of previously-started containers. Document that an operator running the Docker/Apple Container driver in production should assume abandoned sandbox containers from a killed server are **not** automatically cleaned up by this work item (no reconciliation-on-startup pass is in scope) and should rely on `--rm`/equivalent-on-crash host-level hygiene or a periodic external sweep — call this out plainly in `docs/reference/configuration.md` rather than implying a guarantee that does not exist.
- **`AppleContainerDriver` selected on a non-macOS host**: fails fast at startup construction (`SandboxError::Unsupported`), logged once and fatal to server startup if any profile has already declared `available_sandboxes` (there is no point running with a driver that cannot function) — mirrors the "usage error, exit 2" posture `ConfigFileError` already established for other operator-authoring mistakes.
- **A `run_shell_named` command template's interpolated argument contains shell metacharacters** (`;`, `` ` ``, `$( )`, quotes): the shell-escaping step (part D) must neutralize these before the string reaches `docker exec`/`session.execRemoteSandbox`, since a model-supplied argument is untrusted input by definition — this is the primary command-injection surface in this entire work item and must be tested explicitly (see Test Considerations).
- **A `run_shell_command` (full, unconstrained shell) tool bound to a `Local` target**: this is, by design, "run any shell command in this container image" — the harness developer's `image` choice is the entire security boundary (an image with no secrets/host mounts vs. one with them); this work item does not add any command allowlisting to `run_shell_command` itself (that is precisely the difference from `run_shell_named`) — document this distinction prominently so a developer does not mistake `run_shell_command` for a constrained tool.
- **An `Auto`-mode sandbox tool call arrives before `session.startRemoteSandbox` was ever called on that session**: handled identically to a tool with no configured MCP server (`engine/session.rs:409-419` today) — an error-shaped `tool.result`, turn continues, no crash and no silent success.
- **Two drivers on one session, one calls `startRemoteSandbox`**: the started sandbox is session-wide (part C), so a second driver's `Auto`-mode tool calls during *its own* turn can dispatch against the same already-started sandbox without calling `startRemoteSandbox` itself — this is deliberate (mirrors MCP servers being session-wide, not per-driver, per the existing Principle 2 exception) and should be documented as such rather than assumed to be a bug if a harness developer expects per-driver isolation. A future work item could add per-driver sandboxes if that isolation is ever required; out of scope here.
- **`explore_files_tool` traversing a symlink that escapes an `allowed_dirs` root**: rejected by the canonicalize-before-check ordering in `validate_path` (part E) — a symlink inside an allowed directory pointing outside it must not grant access, and this must be covered by a dedicated test (see below), not just asserted in prose.
- **`write_file_tool` asked to write to a path whose parent directory does not exist**: rejected (`create_parents` defaults `false`) rather than silently creating arbitrary intermediate directories under an allowed root, which could otherwise be used to stage files somewhere unexpected within (but still technically inside) the allowlist.
- **client-rust's `ToolHandler` signature change (part D)**: every existing synchronous tool handler in `client-rust`'s own tests/examples/docs (including the `Tool::new` doctest in `tool.rs:32-43` and the `harness.rs` doctest) must be updated to the new async-closure shape in the same change — a partially-migrated crate that compiles some handlers and not others is not an acceptable intermediate state to ship.
- **`session.reportLocalSandbox` is self-reported telemetry, not a security control.** The server cannot verify a client's claim that its local container is actually running/stopped — a client could report `running` for a container that never existed, or never report `stopped` for one that did. This is an accepted trust boundary: local sandboxes are the harness developer's own local resource, and the server's role here is purely to make that activity *visible* in the shared event log for other session participants/subscribers, not to govern or verify it. Document this distinction prominently next to the remote lifecycle's documentation (which *is* authoritative, since the server itself drives it) so an operator does not mistake a `session.sandbox.running` event with `"dispatch": "local"` for something the server has verified.
- **A client harness crashes or disconnects after reporting `session.sandbox.running` for a local sandbox but before ever reporting `stopped`/`error`**: the event log simply has no terminal event for that local sandbox — there is no server-side timeout or reconciliation for client-reported local state (the server holds no local-sandbox handle to reconcile; it never retains one, unlike the remote case where `AppState.sandboxes` is real server-owned state it can positively tear down at session close). This is a known, accepted gap for local sandboxes specifically (the remote lifecycle has no equivalent gap, since the server always eventually logs `stopped`/`error` for its own sandbox at session close) — document it alongside the trust-boundary note above rather than silently leaving a false impression of completeness.
- **A client calls `session.reportLocalSandbox` for a session it is not a registered driver on**: rejected with the same `-32001` gate `sendMessage`/`startRemoteSandbox` already use — reporting local sandbox activity still requires having registered as a driver first, even though it does not require being the current turn's owner.


## Test Considerations:

- **Unit — `SandboxDriver` trait, per implementation**: `ensure_image` is idempotent (a second call on an already-present image is a fast no-op, asserted via call count on a mocked subprocess runner rather than a real Docker daemon — offline per the project's test posture); `start`/`exec`/`stop` map non-zero subprocess exit codes to the correct `SandboxError` variant; `AppleContainerDriver` returns `Unsupported` when constructed with a non-macOS `target_os` override (inject the OS check as a parameter for testability rather than a hard `cfg!` in the constructor body).
- **Unit — image allowlist enforcement, single profile**: `session.startRemoteSandbox` with an image absent from the profile's `available_sandboxes` → `sandbox_image_not_allowed`, no container started (assert the mock driver's `start` was never called).
- **Integration — cross-profile scoping (the key regression this change is about)**: create two profiles, A (`available_sandboxes = ["python:3.12"]`) and B (`available_sandboxes = ["node:22"]`), both provisioned against the same mock driver so `sandbox_status` genuinely holds entries for both. Open a session against profile A and assert **both** of the following in the same test: (1) `session.registerDriver`'s `session.sandbox.available` notification lists only `python:3.12`, never `node:22`, even though the latter is a known, successfully-provisioned image on the same running server; (2) `session.startRemoteSandbox` with `image: "node:22"` against A's session is rejected with `sandbox_image_not_allowed`, and the mock driver's `start` is never called for it. Repeat symmetrically for a session opened against profile B to prove the scoping isn't accidentally one-directional.
- **Integration — background provisioning**: creating a profile with 2 images (one that "pulls" successfully, one that fails) against a mock driver asserts both `tracing::info!`/`tracing::error!` lines are emitted, `sandbox_status` reflects `Available`/`Error` per image, and the admin `POST /admin/v1/profiles` response returns before the mock driver's artificially-delayed pull completes (proving the task is truly backgrounded, not awaited).
- **Integration — driver-connect notification**: a session opened against a profile with `available_sandboxes = ["python:3.12"]`; `session.registerDriver` → assert a `session.sandbox.available` event is broadcast with the correct per-image status, in addition to the existing `session.driver.register` event, and that a profile with an empty `available_sandboxes` emits no such event.
- **Integration — remote sandbox full lifecycle, success path**: `startRemoteSandbox` with an allowed image → assert the ordered pair `session.sandbox.start` then `session.sandbox.running` (both `"dispatch": "remote"`), `sandbox_id` returned in the terminal response; `stopRemoteSandbox` → assert `session.sandbox.stop` then `session.sandbox.stopped`, and `AppState.sandboxes` no longer holds an entry for the session. Session close after a sandbox was started (no explicit stop) → assert the same `stop`/`stopped` pair fires with `"reason": "session_close"` and the mock driver's `stop` was called.
- **Integration — remote sandbox lifecycle, failure paths**: mock driver's `start` fails → assert `session.sandbox.start` then `session.sandbox.error` (`phase: "start"`), `-32012 sandbox_start_failed` returned, no handle retained; mock driver's `stop` fails → assert `session.sandbox.stop` then `session.sandbox.error` (`phase: "stop"`), and the session's `AppState.sandboxes` entry is still removed despite the failure (no phantom handle left behind).
- **Unit — `session.reportLocalSandbox`**: each of `running`/`stopped`/`error` maps to the correct `EventType` with `"dispatch": "local"` in the payload; the caller must be a registered driver (`-32001` otherwise, same gate as `sendMessage`); no validation against `available_sandboxes` is performed (assert an arbitrary/unregistered image name is accepted and logged as-is, proving this path is deliberately not a security gate).
- **Integration — manual remote dispatch**: a client declares an ordinary client tool wrapping remote-manual sandbox exec; the scripted turn produces a `tool_use` for it → assert `Outcome::Paused` (unchanged `run_turn` behavior), then a separate `session.execRemoteSandbox` call against a mock driver returns `{stdout, stderr, exit_code}`, and the client's continuation `session.sendMessage` carries a `tool_result` the harness itself constructed from that output.
- **Integration — auto remote dispatch**: a client declares a `sandbox_tools` entry at session open; the scripted turn's `tool_use` targets it → assert `run_turn` dispatches directly to the mock `SandboxDriver::exec` (never pausing, never reaching the client), logs `sandbox.request`/`sandbox.response`, and the loop continues to a terminal `Completed` outcome in the same turn — cross-check against the existing MCP round-trip integration test's shape (`harness.rs`'s `MCP_PARITY_SEQUENCE` test) since the two dispatch paths should look structurally identical in the event log aside from the event-type names.
- **Integration — auto tool call with no started sandbox**: asserts the existing "no MCP server configured" error-tool-result code path is reused verbatim (same payload shape, `reason` field) rather than a divergent new error format.
- **Unit — command-injection resistance for `run_shell_named`**: a template `echo {name}` called with `name = "a'; rm -rf / #"` (and a handful of other classic shell-metacharacter payloads: backticks, `$()`, `&&`, embedded double quotes) must produce a final command string that, when parsed by a shell, treats the entire argument as **one literal string argument to `echo`** — assert the exact escaped output string per SDK, not just "it didn't crash." This is the single most important test in this work item and must exist in all three SDKs with identical scripted payloads (parity, per the existing cross-SDK convention).
- **Unit — `validate_path` (file tools), shared across all three constructors**: an in-`allowed_dirs` path with an allowed extension → permitted; a path traversal (`../secret.txt`) that resolves outside `allowed_dirs` → rejected; a symlink inside an allowed dir pointing outside it → rejected (create a real temp-dir symlink in the test, don't just assert on the string form); `denied_extensions` overrides an otherwise-matching `allowed_extensions`; `deny_regex` overrides an otherwise-matching `allow_regex`; an empty `allowed_dirs` list rejects every path.
- **Unit — `write_file_tool` parent-directory guard**: a write whose parent directory doesn't exist is rejected when `create_parents` is `false` (default) and succeeds when explicitly enabled.
- **Unit — `explore_files_tool` / `read_file_tool` consistency**: every path `explore_files_tool` returns for a given `allowed_dirs` config must independently pass `read_file_tool`'s own validation under the identical config (a generative/property-style test: for a set of real files scattered both inside and outside the allowed tree, assert the two tools never disagree on a given path's permissibility).
- **Cross-SDK parity**: extend the `harness-smoke` scripted scenarios in all three SDKs to exercise the auto-dispatch sandbox path and the manual-dispatch sandbox path, asserting byte-for-byte identical event-type sequences across Rust/TypeScript/Python for the same scripted mock-driver responses — same convention as the existing `MCP_PARITY_SEQUENCE`/`TWO_DRIVER_PARITY_SEQUENCE` constants in `client-rust/src/harness.rs`'s test module (and their TS/Python mirrors).
- **Regression — `EventType`**: the enum grows from 14 to 22; its exhaustiveness test and `should_broadcast`'s exhaustive match are updated together so a future omission fails to compile.
- **Cross-SDK — local sandbox lifecycle reporting**: in each of the three client SDKs, binding a `Local`-target sandbox tool and driving one scripted tool call against a mock local driver asserts `session.reportLocalSandbox` is called with `state: "running"` before the first `exec`, and `Session::close()` triggers `state: "stopped"` for any local sandbox the harness started — asserted by inspecting the outbound RPC calls the mock transport recorded, the same technique `client-rust/src/harness.rs`'s `MockTransport` already uses for `register_driver` call counting.
- All new tests remain offline: `SandboxDriver` is exercised only through a mock/fake implementation in unit and integration tests (a fake that records calls and returns scripted `ExecResult`s/errors) — no test may require a real `docker`/`container` binary or daemon to be present in CI, mirroring the existing MCP tests' use of a trivial `command = "true"` stdio server rather than a real MCP server.


## Codebase Integration:
- follow established conventions, best practices, testing, and architecture patterns from the project's aspec.
- `engine/sandbox.rs` follows `engine/mcp.rs`'s exact precedent: hand-rolled subprocess-based implementation over an SDK dependency, a closed enum of failure modes, and a lifecycle (`connect`/`start` on demand, explicit `shutdown`/`stop` at session close, `kill_on_drop`-equivalent hygiene where applicable) rather than a new pattern — a reviewer already familiar with `mcp.rs` should recognize `sandbox.rs`'s shape immediately.
- New in-memory per-profile/per-session state (`sandbox_status`, `sandboxes`) belongs on `AppState` alongside `mcp_sessions`/`drivers`/`turn_gates`, uses the same `Arc<Mutex<HashMap<_, _>>>` shape, and is torn down at session close in the same `close` function that already tears down `mcp_sessions`/broadcaster/multi-client runtime state (`api/client/sessions.rs::close`) — add sandbox teardown there, not a separate path.
- Reuse `require_string_array` (`admin/profiles.rs`) verbatim for `available_sandboxes`, exactly as WI 0005's Codebase Integration section already establishes for `fallback_providers`.
- The new `session.startRemoteSandbox`/`session.stopRemoteSandbox`/`session.execRemoteSandbox`/`session.reportLocalSandbox` JSON-RPC methods slot into the existing `rpc()` method match (`api/client/rpc.rs`) beside `registerDriver`/`sendMessage`/`subscribe`/`unsubscribe`, with the same auth/driver-registration gating and the same NDJSON terminal-response framing — no new transport concept. `startRemoteSandbox`/`stopRemoteSandbox` share one internal helper for the stop half so the explicit-stop and close-triggered-stop code paths cannot drift apart (see part C).
- Every place that reads `AppState.sandbox_status` or a profile's `available_sandboxes` to build a client-facing response (the driver-connect notification, `startRemoteSandbox`'s validation, and `GET /admin/v1/sandbox-status`) must scope by the specific `profile_id` in play — never iterate/flatten the full server-wide map — since `sandbox_status` is intentionally one shared map across all profiles for the background-provisioning task's convenience, not because its contents are meant to be cross-profile-visible.
- Keep the sandbox-dispatch concern inside `engine/session.rs::run_turn` structured as a third bucket parallel to the existing client-tool/MCP-tool split, not as a bolted-on special case before/after the existing loop — a future reader should see three symmetric dispatch decisions (client / MCP / sandbox-auto), not two-plus-an-exception.
- Client-SDK builtin tools (`sandbox.rs`/`.ts`/`.py`, `files.rs`/`.ts`/`.py`) are opt-in library code the harness developer explicitly constructs and registers via `Harness::with_tool`/`register_tool` (or each SDK's equivalent) — never auto-registered, consistent with Principle 2 ("thin protocol, customizable harness"): the server and the core harness loop stay agnostic to whether an agent uses these builtins at all.
- The `client-rust` `ToolHandler` async-signature change (part D) is the one genuinely breaking change to an existing public API in this work item; call it out prominently in that crate's changelog/release notes the same way WI 0005 calls out its profile-field-rename breaking change, and update every existing doctest/example alongside the type change in the same commit rather than leaving the crate in a half-migrated state.
- Verify `make image` and `make test` still pass across all four components (`server/`, `client-rust/`, `client-typescript/`, `client-python/`) after this work item; the CI image build should not require a `docker`/`container` binary to be present (all sandbox tests run against a mock driver, per Test Considerations), so `make test` must stay fully offline exactly as it is today.
