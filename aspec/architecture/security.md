# Security

## API Security

Transport:
- TLS terminated at a reverse proxy (nginx/caddy/cloud LB) in front of the server; the container itself speaks plain HTTP on an internal network only. Never expose the server port directly to the internet.
- Local development may use plain HTTP on localhost.

Authentication:
- Bearer API keys (`bae_…` client, `bae_ses_…` session, `bae_admin_…` admin — three distinct roles), generated by the server with sufficient entropy, shown once at creation (client/session) or on first self-generate (admin), and stored only as Argon2id hashes in SQLite, verified in constant time.
- **Admin port (work item 0004).** `/admin/v1/*` requires `Authorization: Bearer <admin_key>` unless the operator explicitly opts out with `--dangerously-disable-admin-auth`. On first `serve` startup the server self-generates a `role='admin'` key and writes its plaintext to `BAE_ADMIN_KEY_FILE` on the data volume (default `/var/lib/bae/admin-key.pem`, `0600` permissions) — **not** printed to stdout/logs, since a file on the data volume can be read programmatically (by `baectl`) and isn't exposed through log aggregation the way a startup log line would be. `--rotate-admin-key` revokes the current admin key and mints a fresh one; a JSON hash file (`BAE_ADMIN_KEY_HASH_FILE`) lets an operator pre-provision one shared admin credential across multiple independently-running replicas without the server ever learning the plaintext. See [Admin authentication](../../docs/guides/admin-authentication.md) and [`baectl`](../../docs/reference/baectl.md) (the recommended way to exercise the admin API) for the full lifecycle.
- LLM provider credentials (e.g. `ANTHROPIC_API_KEY`) and the GitHub MCP credential (`GITHUB_TOKEN`) are supplied via environment variables to whichever process calls the provider/MCP server and are never persisted in the database or echoed in logs. For issue-triage, use the narrowest GitHub scope that works — `issues:write` on the target repository — so the agent cannot push code, alter repository settings, or access secrets it was never given.

RBAC:
- Two key roles to start: `admin` (key management, all resources, server administration) and `agent` (CRUD limited to resources created under that key).
- Enforce least privilege: agent harnesses run with `agent` keys only; `admin` keys are for operators and never embedded in client code.
- Every request is attributed to its key in the event log for auditability.
