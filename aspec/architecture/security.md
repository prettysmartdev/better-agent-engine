# Security

## API Security

Transport:
- TLS terminated at a reverse proxy (nginx/caddy/cloud LB) in front of the server; the container itself speaks plain HTTP on an internal network only. Never expose the server port directly to the internet.
- Local development may use plain HTTP on localhost.

Authentication:
- Bearer API keys (`bae_…` client, `bae_ses_…` session, `bae_admin_…` admin — three distinct roles), generated by the server with sufficient entropy, shown once at creation (client/session) or on first self-generate (admin), and stored only as Argon2id hashes in SQLite, verified in constant time.
- **Admin port (work item 0004).** `/admin/v1/*` requires `Authorization: Bearer <admin_key>` unless the operator explicitly opts out with `--dangerously-disable-admin-auth`. On first `serve` startup the server self-generates a `role='admin'` key and writes its plaintext to `BAE_ADMIN_KEY_FILE` on the data volume (default `/var/lib/bae/admin-key.pem`, `0600` permissions) — **not** printed to stdout/logs, since a file on the data volume can be read programmatically (by `baectl`) and isn't exposed through log aggregation the way a startup log line would be. `--rotate-admin-key` revokes the current admin key and mints a fresh one; a JSON hash file (`BAE_ADMIN_KEY_HASH_FILE`) lets an operator pre-provision one shared admin credential across multiple independently-running replicas without the server ever learning the plaintext. See [Admin authentication](../../docs/guides/09-admin-authentication.md) and [`baectl`](../../docs/reference/03-baectl.md) (the recommended way to exercise the admin API) for the full lifecycle.
- LLM provider credentials (e.g. `ANTHROPIC_API_KEY`) and the GitHub MCP credential (`GITHUB_TOKEN`) are supplied via environment variables to whichever process calls the provider/MCP server and are never persisted in the database or echoed in logs. For issue-triage, use the narrowest GitHub scope that works — `issues:write` on the target repository — so the agent cannot push code, alter repository settings, or access secrets it was never given.
- **OpenTelemetry (work item 0013).** The same "never logged" guarantee above extends explicitly to telemetry: the admin key, client/session keys, and resolved provider/MCP `${VAR}` secrets (including resolved `[telemetry].otlp_headers` values) must never be logged **nor exported as span attributes, span events, span/metric names, links, or metric labels**. Spans may carry opaque ids (session id, profile id, client-key id, subagent id), tool/MCP-server/provider names, and size/shape metadata (byte counts, block counts) — never key material, never tool input/output payload content, message text, or provider request/response bodies. Metric labels are restricted to a small closed set of enum-like values (e.g. `outcome`, `dispatch`, `provider` kind) and must never be a high-cardinality value like a session id, tool name, or MCP server name. A regression test asserts none of a fixture's real secrets appear anywhere in exported span/metric payloads. See [Configuration — `[telemetry]`](../../docs/reference/05-configuration.md#telemetry).

RBAC:
- Two key roles to start: `admin` (key management, all resources, server administration) and `agent` (CRUD limited to resources created under that key).
- Enforce least privilege: agent harnesses run with `agent` keys only; `admin` keys are for operators and never embedded in client code.
- Every request is attributed to its key in the event log for auditability.
